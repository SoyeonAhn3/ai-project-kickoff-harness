AI Project Kickoff Harness 기획 정리

이 프로젝트는 사용자가 막연한 아이디어만 가지고 있어도, AI가 질문을 통해 요구사항을 끌어내고, 프로젝트를 실제 개발 가능한 상태까지 정리해주는 개발 착수 전 자동화 하네스다.

단순히 기획서를 대신 써주는 도구가 아니라, 프로젝트를 시작하기 전에 필요한 질문, 의사결정, 문서화, 개발 단계 설계를 자동으로 도와주는 시스템으로 만든다.

핵심 컨셉

사용자가 "이런 프로젝트를 만들고 싶다" 정도만 입력하면, 하네스가 프로젝트 유형을 파악하고 필요한 질문을 던진다.

사용자의 답변을 바탕으로 요구사항을 정리하고, 부족한 부분이나 모순되는 부분을 찾아낸 뒤, 개발에 필요한 기본 산출물을 자동으로 만든다.

최종적으로 사용자는 바로 개발에 들어가기 전에 필요한 문서들을 갖게 된다.

예를 들면:

프로젝트 아이디어
→ AI 질문
→ 사용자 답변
→ 요구사항 정리
→ 누락/모순 점검
→ 개발 착수 체크리스트 생성
→ 설계서 생성
→ 데이터 구조 문서 생성
→ Readme, Phase 문서 생성

이 하네스의 목적

이 하네스의 목적은 코드를 바로 만들어주는 것이 아니다.

목적은 개발 전에 프로젝트를 명확하게 정의하는 것이다.

많은 프로젝트가 실패하거나 꼬이는 이유는 코드 때문이 아니라, 개발 전에 아래 내용이 명확하지 않기 때문이다.

무엇을 만들 것인가
누구를 위한 것인가
어떤 문제를 해결하는가
MVP 범위는 어디까지인가
어떤 데이터가 필요한가
AI는 어디에 쓰이는가
어떤 기술 스택을 쓸 것인가
실패 상황은 어떻게 처리할 것인가
어떤 문서를 기준으로 개발할 것인가

이 하네스는 이런 부분을 AI가 질문하고 정리하면서, 프로젝트가 흐릿한 아이디어에서 개발 가능한 계획으로 바뀌도록 돕는다.

기존 스킬 패키지와의 관계

이미 만들어둔 스킬들은 이 하네스 안에서 각각의 기능 모듈처럼 사용한다.

하네스가 전체 흐름을 관리하고, 각 단계에서 필요한 스킬을 호출하는 구조다.

즉, 하네스는 새로운 단일 스킬이라기보다 기존 스킬들을 묶어서 프로젝트 시작 과정을 자동화하는 상위 오케스트레이터에 가깝다.

전체 흐름

하네스는 크게 이런 흐름으로 동작한다.

1. 아이디어 입력

사용자가 만들고 싶은 프로젝트를 간단히 설명한다.

예:

생산 데이터를 보고 이상을 감지하고 AI가 원인을 분석해서 알려주는 시스템을 만들고 싶다.

2. 프로젝트 유형 판별

AI가 프로젝트 유형을 먼저 파악한다.

예:

AI 프로젝트
자동화 프로젝트
BI 대시보드 프로젝트
데이터 파이프라인 프로젝트
웹앱 프로젝트
CLI 도구
라이브러리

프로젝트 유형에 따라 필요한 질문과 문서 구조가 달라진다.

3. 요구사항 인터뷰 (2단계 구성)

인터뷰는 기획과 설계 2단계로 나뉜다. 사용자가 기획만 하고 싶으면 설계 인터뷰를 "패스"할 수 있다.

공통 인터뷰 규칙:
- 질문 방식: 묶음 질문(3~5개) + 부족하면 추가 질문
- "잘 모르겠어" / "추천해줘"를 정식 답변으로 인정
- 사용자가 아는 게 있으면 본인 답변 우선, 모르면 AI 추천 채택
- 사소한 디테일은 묻지 않음
- AI 추천 규칙: 사용자가 이미 밝힌 목적/맥락에 맞는 선택지를 우선 추천한다. "쉬운 것"이 아니라 "목적에 맞는 것"이 먼저.

--- Step 3-1: 기획 인터뷰 ---

"무엇을 만들 것인가"에 집중. 비기술적 사용자도 답변 가능한 수준.

질문 예:
- 이 프로젝트의 주요 사용자는 누구인가?
- 어떤 문제를 해결하는가?
- 입력 데이터는 어디서 오는가?
- 결과는 어떤 형태로 보고 싶은가?
- MVP 범위는 어디까지인가?

--- Step 3-2: 설계 인터뷰 ---

"어떻게 만들 것인가"에 집중. 기획 인터뷰 답변을 기반으로 기술적 구조를 구체화.
사용자가 "패스"하면 이 단계를 건너뛰고 Step 3.5로 넘어간다.

질문 영역:

1. 파이프라인 흐름
   - 데이터가 소스에서 최종 사용자에게 도달하기까지의 단계별 흐름
   - 모르면 AI가 프로젝트 유형 기반으로 추천

2. 레이어별 테이블/데이터 구조
   - 각 레이어(Raw/Warehouse/Mart 등)에 어떤 테이블이 필요한지
   - 주요 컬럼을 AI가 추천하고 왜 그 컬럼이 필요한지 설명
   - 사용자가 수정/채택/거절 선택

3. 데이터 볼륨 및 갱신 주기
   - 예상 데이터 크기, 갱신 빈도
   - 사용자 답변 또는 AI 추천

4. 실패 시나리오 / 엣지케이스 (가볍게)
   - "이 시스템이 실패할 수 있는 상황은?" 한 질문으로 수집
   - 모르면 AI가 프로젝트 유형 기반으로 3~5개 추천
   - 킥오프에서는 인식 수준만. 상세 시나리오는 개발 단계(test-scenario 스킬)에서 작성

3.5 AI 아이디어 제안

인터뷰 완료 후, AI가 프로젝트 유형과 사용자 답변을 기반으로 추가 아이디어를 제안한다.

동작 방식:
- AI가 최대 5개의 추가 아이디어를 제안
- 각 아이디어에 "왜 유용한지" 한 줄 근거 포함
- 각 아이디어에 개발 난이도(낮/중/높)와 예상 시간도 함께 제공
- 사용자가 채택 / 거절 / 수정 / 전부 패스 선택
- 채택된 아이디어만 프로필에 반영 (MVP 범위 또는 "MVP 이후 고려사항"으로 분류 가능)

주의:
- 최대 5개로 제한 (scope creep 방지)
- "전부 패스"를 명시적 선택지로 제공
- 제안은 강제가 아닌 참고용임을 명시

4. 프로젝트 프로필 생성 → project_kickoff.md 파일 생성

인터뷰 답변 + 채택된 아이디어를 구조화하여 project_kickoff.md 파일을 생성한다.
이 파일은 이후 모든 문서와 스킬 실행의 기준이 된다.
이후 Step 5, 8에서 이 파일에 섹션을 추가하는 방식으로 문서가 완성된다.

project_kickoff.md 최종 구조:

  1. 프로젝트 프로필 (기본정보, 목적, 사용자, 핵심기능, 기술스택, 제약사항) ← Step 4에서 작성
  2. 시스템 아키텍처 (파이프라인 흐름, 컴포넌트 구성)                        ← Step 4에서 작성 (3-2 답변 기반)
  3. 데이터 구조 (레이어별 테이블, 컬럼, 볼륨, 갱신주기)                     ← Step 4에서 작성 (3-2 답변 기반)
  4. 실패 시나리오 / 엣지케이스                                               ← Step 4에서 작성 (3-2 답변 기반)
  5. 누락/모순 점검 결과                                                      ← Step 5에서 추가
  6. 개발 착수 체크리스트                                                     ← Step 8에서 추가

설계 인터뷰(3-2)를 패스한 경우 섹션 2, 3, 4는 생략된다.

파일 생성 방식:
- Step 4 (kickoff-profile): project_kickoff.md 파일 생성, 섹션 1~4 작성
- Step 5 (kickoff-gap): 기존 파일을 읽고 섹션 5를 하단에 추가
- Step 8 (kickoff-checklist): 기존 파일을 읽고 섹션 6을 하단에 추가

저장 위치: {현재 디렉토리}/pre-requirement/project_kickoff.md

5. 누락/모순 점검

AI가 project_kickoff.md의 내용을 검토해서 빠진 부분이나 충돌하는 부분을 찾는다.
결과를 project_kickoff.md 섹션 5에 추가한다.

점검 규칙:
- 명확한 모순/오류만 잡는다
- 애매한 건 사용자에게 질문해서 사용자가 결정한다
- 해결된 내용은 "문제 + 해결" 형태로 기록

예:

데이터 저장소는 정했지만 중복 방지 방식이 없음
AI 결과가 틀렸을 때의 폴백이 없음
스케줄러 실행 시간과 데이터 갱신 시간 충돌

이 단계가 중요하다.
이 하네스는 단순 문서 생성기가 아니라, 개발 전에 문제를 찾아내는 도구가 되어야 한다.

8. 개발 착수 체크리스트 생성

project_kickoff.md의 프로필/아키텍처 내용을 기반으로 체크리스트를 생성한다.
결과를 project_kickoff.md 섹션 6에 추가한다.

예:

기술 스택 확정 여부
데이터 소스 확정 여부
환경변수 목록 정리 여부
API 키 필요 여부
샘플 데이터 준비 여부
MVP 범위 확정 여부

이 체크리스트를 통과하면 개발을 시작할 수 있는 상태가 된다.

이 하네스의 핵심 차별점

이 프로젝트의 차별점은 단순히 문서를 생성하는 것이 아니다.

핵심은 아래 4가지다.

1. 질문을 통해 사용자의 요구사항을 끌어낸다.
2. 답변을 구조화된 프로젝트 프로필로 만든다.
3. 누락된 결정과 문서 간 모순을 찾아낸다.
4. 기존 스킬들을 연결해 개발 착수 전 산출물을 자동 생성한다.

즉, 이 하네스는 AI 문서 생성기가 아니라
AI 기반 프로젝트 착수 시스템이다.
