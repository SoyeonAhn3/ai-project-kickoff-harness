# design-architecture Skill

킥오프 문서(섹션 1~9)를 분석하여 시스템 구조, 기술 선택 근거, 배포 구성, 횡단 관심사, 아키텍처 결정 기록(ADR)을 생성하고 `_kickoff.md`의 섹션 10으로 추가한다.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read
  - 2개 이상 발견: 사용자에게 선택 요청
- YAML frontmatter `completed_skills`에 `design-requirements`가 포함되어야 함
  - 아니면 `/design-requirements`를 먼저 실행하도록 안내
- 섹션 9(요구사항 정의)가 존재해야 함
  - 없으면 `/design-requirements`를 먼저 실행하도록 안내

## STEP 1 — 킥오프 문서 분석

킥오프 문서의 섹션 1~9를 Read하여 아래 정보를 추출한다:

| 추출 항목 | 위치 | 용도 |
|---|---|---|
| 프로젝트 유형 | 섹션 1 → 유형 | 아키텍처 패턴 선택 |
| 프로젝트 목적 | 섹션 1 → 프로젝트 배경 | 배포 섹션 활성화 판정 |
| 주요 사용자 | 섹션 1 → 내부/외부 | 배포 섹션 활성화 판정 |
| 기술 스택 | 섹션 1 → 기술 스택 | 기술 선택 근거 확장 |
| 기존 아키텍처 | 섹션 2 (있는 경우) | 확장 기반 |
| 기능 요구사항 | 섹션 9-1 | 컴포넌트 도출 |
| 비기능 요구사항 | 섹션 9-2 | 횡단 관심사 도출 |
| 사용자 흐름 | 섹션 9-3 | 시스템 흐름 매핑 |

### 재실행 판정

섹션 10이 이미 존재하는 경우:
- 사용자에게 알림: "섹션 10이 이미 존재합니다. 덮어쓸까요, 아니면 수정할까요?"
- **덮어쓰기**: 기존 섹션 10을 삭제 후 새로 생성
- **수정**: 사용자가 지정한 부분만 Edit

## STEP 2 — 시스템 구조 생성

`references/architecture-patterns.md`에서 프로젝트 유형에 맞는 패턴을 참조하여 시스템 구조를 생성한다.

### 2-1. 아키텍처 패턴 선택

1. 유형별 패턴 카탈로그에서 후보 패턴 2~3개 제시
2. 각 패턴의 적합도를 프로젝트 컨텍스트 기반으로 평가
3. 추천 패턴 1개 선택 (사용자 확인)

### 2-2. 시스템 다이어그램

`references/architecture-template.md`의 다이어그램 형식을 참조하여 텍스트 기반 시스템 다이어그램을 작성한다.

### 2-3. 컴포넌트 테이블

섹션 9-1(FR)에서 도출한 컴포넌트를 테이블로 정리:

```
| 컴포넌트 | 역할 | 기술 | 입력 | 출력 |
|---|---|---|---|---|
```

### 2-4. 통신 테이블

컴포넌트 간 통신 방식을 정리:

```
| 소스 | 대상 | 방식 | 프로토콜 |
|---|---|---|---|
```

### 섹션 2와의 관계

- 섹션 2가 있는 경우: 섹션 2의 파이프라인 흐름/외부 의존성을 기반으로 **확장**
- 섹션 2가 없는 경우 (설계 패스): 섹션 10이 아키텍처의 유일한 출처로 작성

## STEP 3 — 기술 선택 근거 생성

섹션 1 기술 스택 테이블을 확장하여 선택 근거를 상세화한다.

### 규칙

1. 섹션 1의 각 레이어에 대해 비교 대안 최소 1개 제시
2. "AI 추천"으로 표기된 기술은 추천 근거를 구체적으로 보충
3. 선택 이유는 프로젝트 컨텍스트에 맞춰 구체적으로 기술

### 출력 형식

```
### 10-2. 기술 선택 근거 (Tech Selection Rationale)

| 레이어 | 선택 기술 | 비교 대안 | 선택 이유 |
|---|---|---|---|
| [레이어명] | [기술명] | [대안들] | [구체적 근거] |
```

## STEP 4 — 배포 구성 생성 (조건부)

`references/architecture-template.md`의 배포 활성화 규칙에 따라 판정한다.

### 활성화 조건

| 조건 | 배포 섹션 |
|---|---|
| 목적 = 실무 투입 | 전체 활성 (dev/staging/prod) |
| 사용자 = 외부 포함 | 전체 활성 |
| 목적 = 포트폴리오 | dev + prod만 |
| 목적 = 학습 | 비활성 |
| 유형 = CLI/라이브러리 | 패키지 배포로 대체 |

### 비활성 시 출력

```
### 10-3. 배포 구성 (Deployment)

> 배포 구성은 실무/외부 서비스 프로젝트에서 활성화됩니다.
```

### 활성 시 출력

```
### 10-3. 배포 구성 (Deployment)

| 환경 | 구성 | 설명 |
|---|---|---|
| 개발 (dev) | [구성] | [설명] |
| 프로덕션 | [구성] | [설명] |
```

## STEP 5 — 아키텍처 인터뷰

STEP 2~4 결과를 바탕으로 불명확한 아키텍처 결정에 대해 사용자에게 질문한다.

### 규칙

- **최대 5개 질문** (사용자 부담 최소화)
- 질문은 구체적이고 선택지를 포함
- 이미 킥오프 문서에 답이 있는 질문은 하지 않음
- 답변 결과를 아키텍처에 반영

### 질문 예시

| 상황 | 질문 예시 |
|---|---|
| DB 미정 | "데이터 특성상 RDB(PostgreSQL)와 NoSQL(MongoDB) 중 어떤 것이 적합할까요?" |
| 배포 방식 미정 | "Docker 컨테이너 vs 서버리스(AWS Lambda) 중 선호하는 방식이 있나요?" |
| 캐싱 필요 여부 | "NFR의 응답 시간 목표(p95 < 500ms)를 위해 캐싱 레이어를 추가할까요?" |

### 출력 형식

```
## 📋 아키텍처 확인 질문

| # | 질문 | 관련 영역 | 선택지 |
|---|---|---|---|
| 1 | [질문] | 시스템 구조/기술 선택/배포 | A / B / 기타 |
```

## STEP 6 — 섹션 10 초안 제시 및 확인

STEP 2~5 결과를 조합하여 섹션 10 전체 초안을 사용자에게 제시한다.

### 횡단 관심사 생성

`references/architecture-patterns.md`의 공통 횡단 관심사를 참조하되, 섹션 9-2(NFR)에서 도출된 요구사항을 반영:

```
### 10-4. 횡단 관심사 (Cross-cutting Concerns)

| 관심사 | 접근법 | 비고 |
|---|---|---|
| 로깅 | [방식] | [NFR 연계] |
| 에러 처리 | [방식] | [NFR 연계] |
| 설정 관리 | [방식] | |
| 테스트 전략 | [방식] | [NFR-M02 커버리지 목표 연계] |
```

### ADR 생성

아키텍처 패턴 선택, 주요 기술 선택 등에 대해 ADR을 작성:

```
### 10-5. 아키텍처 결정 기록 (ADR)

#### ADR-001: [결정 제목]

**상태**: 채택
**맥락**: [왜]
**결정**: [무엇]
**근거**: [왜 이것]
**결과**: [영향]
```

### 확인 요청

```
### 확인 요청

1. 이대로 확정
2. 수정 요청 (섹션/항목 지정)
3. 아키텍처 패턴 변경
```

## STEP 7 — 파일 업데이트

### 7-1. 섹션 10 추가

킥오프 문서의 섹션 9 뒤에 확정된 섹션 10을 추가한다.

- 섹션 10이 없으면: 섹션 9 끝 이후에 추가 (Write/Edit)
- 섹션 10이 있으면 (재실행): 기존 섹션 10을 Edit으로 교체

### 7-2. YAML frontmatter 갱신

```yaml
last_skill: design-architecture
completed_skills:
  - ... (기존 목록)
  - design-architecture
```

### 7-3. Revision History 추가

```
| [오늘 날짜] | 섹션 10 | 시스템 아키텍처 상세 최초 작성 (컴포넌트 N개, ADR N개) | design-architecture |
```

### 7-4. 교차 섹션 수정 (해당 시)

STEP 5 인터뷰 결과로 기존 섹션(1~9)을 수정해야 하는 경우:
1. 사용자에게 수정 여부 확인
2. 승인 시: 해당 섹션 Edit + Revision History에 기록
3. 미승인 시: 섹션 10-5 ADR에 메모로 기록

- ADR: N개
- 교차 수정: [있음/없음]

## 주의사항

- 모든 출력은 `_kickoff.md` 단일 파일에 추가 (별도 파일 생성 금지)
- 섹션 2(기존 아키텍처)를 대체하지 않고 공존 — 섹션 10은 확장 버전
- 기술 선택 근거는 섹션 1의 "선택 이유"를 확장하되, 섹션 1 자체를 수정하지 않음
- 배포 구성은 조건부 활성화 — 비활성 사유를 명시
- ADR은 트레이드오프가 있는 결정에만 작성 (자명한 선택에는 불필요)
- 교차 섹션 수정 시 반드시 사용자 확인 후 진행 + Revision History 기록
