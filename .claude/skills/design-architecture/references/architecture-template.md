# Architecture Template

`design-architecture` 스킬이 섹션 10(시스템 아키텍처 상세)을 `_kickoff.md`에 작성할 때 참조하는 구조 가이드.
80% 공통 구조 + 20% 프로젝트 유형별 오버라이드로 구성된다.

---

## 섹션 2와의 관계

- **섹션 2 (시스템 아키텍처)**: v1 설계 인터뷰에서 작성한 기초 아키텍처 (파이프라인 흐름, 외부 의존성)
- **섹션 10 (시스템 아키텍처 상세)**: v2 설계 스킬이 섹션 9(요구사항)를 반영하여 확장한 상세 아키텍처
- 두 섹션은 **공존** — 섹션 10이 섹션 2를 대체하지 않음
- 섹션 2가 없는 경우 (설계 인터뷰 패스): 섹션 10이 아키텍처의 유일한 출처

## 시스템 다이어그램 형식

텍스트 기반으로 시스템 구조를 표현한다. 프로젝트 유형에 따라 적합한 형식을 선택:

### ASCII 다이어그램 (기본)

```
[사용자] → [Frontend] → [API Server] → [DB]
                                     → [External API]
```

### 계층형 (Layered)

```
┌─────────────────────────────┐
│       Presentation          │
├─────────────────────────────┤
│       Business Logic        │
├─────────────────────────────┤
│       Data Access           │
├─────────────────────────────┤
│       Infrastructure        │
└─────────────────────────────┘
```

### 파이프라인형 (Data Flow)

```
[Source] → [Ingest] → [Transform] → [Store] → [Serve]
```

## 유형별 구조 오버라이드 (20%)

### 웹앱

- 10-1: 프론트엔드/백엔드/DB 3계층 + 외부 서비스
- 10-2: 프레임워크 선택 이유 강조 (React vs Vue, Express vs FastAPI 등)
- 10-3: 활성 (대부분 배포 필요)
- 10-4: CORS, 인증 미들웨어, 세션 관리 추가

### 데이터 파이프라인

- 10-1: 소스 → 수집 → 변환 → 적재 → 서빙 흐름
- 10-2: 오케스트레이터 선택 (Airflow vs Prefect 등)
- 10-3: 스케줄러 설정, 재처리 환경
- 10-4: 데이터 검증, 스키마 진화, 모니터링 강조

### AI

- 10-1: 학습 파이프라인 + 서빙 파이프라인 분리
- 10-2: 모델/프레임워크 선택 + 인프라 (GPU/CPU)
- 10-3: 모델 레지스트리, A/B 테스트 배포 (해당 시)
- 10-4: 실험 추적, 모델 버저닝, 폴백 전략

### 자동화

- 10-1: 트리거 → 처리 → 액션 → 알림 흐름
- 10-2: 스케줄러/이벤트 소스 선택
- 10-3: 실행 환경 (cron/서버리스/컨테이너)
- 10-4: 멱등성, 재시도, 동시성 관리

### BI

- 10-1: 데이터 소스 → (ETL) → 분석 저장소 → 대시보드
- 10-2: BI 도구 선택 (Tableau/PowerBI/Metabase 등)
- 10-3: 데이터 새로고침 스케줄
- 10-4: 캐싱, 접근 제어, 지표 일관성

### CLI

- 10-1: 진입점 → 파서 → 커맨드 → 출력 구조
- 10-2: CLI 프레임워크 선택 (Click/Typer/Commander 등)
- 10-3: 비활성 (대부분 로컬 실행)
- 10-4: 플러그인 시스템 (해당 시), 설정 파일 로딩

### 라이브러리

- 10-1: 공개 API → 내부 모듈 → 유틸리티 계층
- 10-2: 의존성 최소화 원칙
- 10-3: 패키지 배포 (PyPI/npm 등)
- 10-4: 하위 호환, 시맨틱 버저닝, 변경 이력

## 기술 선택 근거 작성 규칙

1. 섹션 1 기술 스택의 "선택 이유" 컬럼을 **확장**한다 (대체하지 않음)
2. 각 레이어에 최소 1개 비교 대안을 제시
3. 선택 이유는 프로젝트 컨텍스트에 맞춰 구체적으로 기술
   - 좋은 예: "팀 내 Python 경험이 풍부하고, FastAPI의 자동 OpenAPI 문서가 프론트 연동에 유리"
   - 나쁜 예: "인기 있는 프레임워크라서"
4. "AI 추천"으로 표기된 기술은 추천 근거를 구체적으로 보충

## 배포 구성 활성화 규칙

| 조건 | 배포 섹션 |
|---|---|
| 목적 = 실무 투입 | 전체 활성 (dev/staging/prod) |
| 사용자 = 외부 포함 | 전체 활성 |
| 목적 = 포트폴리오 | dev + prod만 (staging 생략 가능) |
| 목적 = 학습 | 비활성 |
| 유형 = CLI/라이브러리 | 패키지 배포로 대체 |
