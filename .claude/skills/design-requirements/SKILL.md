# design-requirements Skill

킥오프 문서(섹션 1~8)를 분석하여 기능 요구사항(FR), 비기능 요구사항(NFR), 사용자 흐름, 제약사항 보강을 생성하고 `_kickoff.md`의 섹션 9로 추가한다.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read
  - 2개 이상 발견: 사용자에게 선택 요청
- YAML frontmatter `completed_skills`에 `kickoff-done`이 포함되어야 함
  - 아니면 `/kickoff-done`을 먼저 실행하도록 안내
- 기술 스택에 "미정" 항목이 있으면 **반드시 사용자에게 확정을 요청**한 후 진행
  - 미정 레이어를 표로 보여주고, 추천 + 사용자 선택을 받아 섹션 1 기술 스택 테이블을 Edit

---

## STEP 1 — 킥오프 문서 분석

킥오프 문서의 섹션 1~8을 Read하여 아래 정보를 추출한다:

| 추출 항목 | 위치 |
|---|---|
| 프로젝트 유형 | 섹션 1 → 기본 정보 → 유형 |
| 프로젝트 목적 | 섹션 1 → 프로젝트 배경 → 프로젝트 목적 |
| 주요 사용자 (내부/외부) | 섹션 1 → 기본 정보 |
| 핵심 기능 테이블 | 섹션 1 → 핵심 기능 |
| 채택된 AI 제안 | 섹션 1 → 채택된 AI 제안 |
| 기술 스택 | 섹션 1 → 기술 스택 |
| 제약사항 | 섹션 1 → 제약사항 |
| 시스템 아키텍처 | 섹션 2 (있는 경우) |
| 데이터 구조 | 섹션 3 (있는 경우) |
| 실패 시나리오 | 섹션 4 (있는 경우) |
| 완료 조건 | 섹션 8 |

### 재실행 판정

섹션 9가 이미 존재하는 경우:
- 사용자에게 알림: "섹션 9가 이미 존재합니다. 덮어쓸까요, 아니면 수정할까요?"
- **덮어쓰기**: 기존 섹션 9를 삭제 후 새로 생성
- **수정**: 사용자가 지정한 부분만 Edit

## STEP 2 — 기능 요구사항 (FR) 생성

`references/requirements-template.md`의 FR 생성 규칙에 따라 생성한다.

### 2-1. 공통 FR 도출

1. 섹션 1 핵심 기능 테이블의 각 행에서 FR 도출
2. 채택된 AI 제안 중 MVP 분류 항목에서 FR 도출
3. 출처 컬럼에 도출 근거 기록

### 2-2. 유형별 추가 FR

`requirements-template.md`의 유형별 추가 FR 영역을 참조하여, 해당 프로젝트 유형에 맞는 추가 FR을 생성한다.

### 출력 형식

```
### 9-1. 기능 요구사항 (Functional Requirements)

| ID | 기능 | 설명 | 우선순위 | 출처 |
|---|---|---|---|---|
| FR-001 | [기능명] | [구체적 동작] | 필수/선택 | 섹션 1 기능#N |
| FR-002 | ... | ... | ... | ... |
```

## STEP 3 — 비기능 요구사항 (NFR) 생성

`references/nfr-checklist.md`의 활성화 판정 규칙에 따라 생성한다.

### 3-1. 카테고리 활성화 판정

킥오프 문서에서 추출한 유형/목적/사용자 정보로 7개 카테고리의 활성/비활성을 판정한다.

### 3-2. NFR 테이블 생성

활성 카테고리의 항목 + 유형별 조정 사항을 반영하여 NFR 테이블을 생성한다.
메트릭은 구체적 수치로 기본값을 제시한다.

### 3-3. 비활성 카테고리 기록

비활성 카테고리는 사유 템플릿을 채워 별도 테이블에 기록한다.

### 출력 형식

```
### 9-2. 비기능 요구사항 (Non-Functional Requirements)

| ID | 카테고리 | 항목 | 목표 메트릭 | 우선순위 |
|---|---|---|---|---|
| NFR-001 | 성능 | 응답 시간 | API p95 < 500ms | 높음 |
| NFR-002 | ... | ... | ... | ... |

**비활성 카테고리:**
| 카테고리 | 비활성 사유 |
|---|---|
| 보안 | 외부 사용자 없는 포트폴리오 프로젝트로, 보안 요구사항 해당 없음 |
```

## STEP 4 — 사용자 흐름 생성

`requirements-template.md`의 사용자 흐름 생성 규칙에 따라 생성한다.

1. 핵심 기능의 주요 흐름 (happy path) 작성
2. 대안 흐름 (조건 분기) 작성
3. 예외 흐름 (에러 처리) 작성 — 섹션 4가 있으면 반영
4. 유형에 맞는 흐름 표현 방식 사용

### 출력 형식

```
### 9-3. 사용자 흐름 (User Flows)

#### 주요 흐름: [흐름명]

1. [사용자/시스템 행위 1]
2. [사용자/시스템 행위 2]
3. ...

#### 대안 흐름

- [조건]: [분기 동작]

#### 예외 흐름

- [에러 조건]: [에러 처리]
```

---

## STEP 5 — 요구사항 갭 인터뷰

STEP 2~4 결과를 바탕으로 불명확하거나 누락된 요구사항에 대해 사용자에게 질문한다.

### 규칙

- **최대 5개 질문** (사용자 부담 최소화)
- 질문은 구체적이고 선택지를 포함 (예: "A 방식 / B 방식 중 어떤 걸 선호하세요?")
- 이미 킥오프 문서에 답이 있는 질문은 하지 않음
- 답변 결과를 FR/NFR/흐름에 반영

### 출력 형식

```
## 📋 요구사항 확인 질문

아래 항목이 불명확하여 확인이 필요합니다:

| # | 질문 | 관련 영역 | 선택지 |
|---|---|---|---|
| 1 | [질문] | FR/NFR/흐름 | A / B / 기타 |
| 2 | ... | ... | ... |
```

---

## STEP 6 — 섹션 9 초안 제시 및 확인

STEP 2~5 결과를 조합하여 섹션 9 전체 초안을 사용자에게 제시한다.

### 출력 형식

```
## 📐 섹션 9 — 요구사항 정의 (초안)

[섹션 9 전체 내용]

---

### 확인 요청

1. 이대로 확정
2. 수정 요청 (번호/항목 지정)
3. FR/NFR 추가 요청
```

### 수정 처리

- 사용자가 수정 요청 시 해당 부분만 수정 후 다시 제시
- 확정될 때까지 반복

---

## STEP 7 — 파일 업데이트

### 7-1. 섹션 9 추가

킥오프 문서의 섹션 8 뒤에 확정된 섹션 9를 추가한다.

- 섹션 9가 없으면: 섹션 8 끝 이후에 추가 (Write/Edit)
- 섹션 9가 있으면 (재실행): 기존 섹션 9를 Edit으로 교체

### 7-2. YAML frontmatter 갱신

```yaml
last_skill: design-requirements
completed_skills:
  - ... (기존 목록)
  - design-requirements
```

### 7-3. Revision History 추가

```
| [오늘 날짜] | 섹션 9 | 요구사항 정의 최초 작성 (FR N개, NFR N개) | design-requirements |
```

### 7-4. 교차 섹션 수정 (해당 시)

STEP 5 인터뷰 결과로 기존 섹션(1~8)을 수정해야 하는 경우:
1. 사용자에게 수정 여부 확인
2. 승인 시: 해당 섹션 Edit + Revision History에 기록
3. 미승인 시: 섹션 9-4 제약사항 보강에 메모로 기록

### 완료 출력

```
[design-requirements] 완료
- 대상 파일: [킥오프 문서 경로]
- FR: N개 (필수 N, 선택 N)
- NFR: N개 (활성 카테고리 N, 비활성 N)
- 사용자 흐름: N개
- 교차 수정: [있음/없음]
- 다음 단계: /design-architecture
```

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-start`를 먼저 실행하도록 안내 |
| `completed_skills` 미달 | `/kickoff-done`을 먼저 실행하도록 안내 |
| 기술 스택 "미정" 존재 | 미정 항목 표시 + 추천 제시 + 사용자 확정 요청 |
| 섹션 9 이미 존재 | 덮어쓰기/수정 선택지 제시 |
| 파일 Edit 실패 | 해당 섹션 다시 Read 후 정확한 문자열로 재시도 |

---

## 주의사항

- 모든 출력은 `_kickoff.md` 단일 파일에 추가 (별도 파일 생성 금지)
- FR/NFR ID는 중복 없이 순번 부여
- NFR 메트릭은 반드시 구체적 수치 제시 (사용자가 조정 가능)
- 킥오프 문서에서 이미 확정된 내용과 모순되는 요구사항을 생성하지 않음
- 교차 섹션 수정 시 반드시 사용자 확인 후 진행 + Revision History 기록
- 재실행 시 기존 섹션 9를 Edit (전체 재작성 아님, 사용자가 덮어쓰기 선택 시 제외)
