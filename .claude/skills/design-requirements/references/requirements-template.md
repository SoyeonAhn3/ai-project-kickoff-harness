# Requirements Template

`design-requirements` 스킬이 섹션 9(요구사항 정의)를 생성할 때 참조하는 템플릿.
80% 공통 구조 + 20% 프로젝트 유형별 오버라이드로 구성된다.

---

## 작성 원칙

1. **측정 가능**: 각 요구사항에 검증 가능한 기준을 포함한다 (수치, 조건, 범위)
2. **테스트 가능**: 완료 여부를 테스트로 판단할 수 있어야 한다
3. **모호성 금지**: "빠른", "좋은", "적절한" 등 주관적 표현을 구체적 수치로 교체
4. **단일 책임**: 하나의 FR/NFR은 하나의 요구사항만 기술
5. **추적 가능**: 킥오프 문서 섹션 1 핵심 기능 테이블과 1:N 매핑

## FR 생성 규칙

### 공통 (80%)

1. 섹션 1 핵심 기능 테이블의 각 행에서 1개 이상의 FR 생성
2. 채택된 AI 제안(섹션 1)에서 MVP 분류된 항목도 FR 생성
3. **출처 컬럼**: 어디서 도출되었는지 기록 (예: "섹션 1 기능#2", "제안#1")
4. **우선순위**: MVP 포함 = 필수, MVP 미포함 = 선택
5. ID 체계: FR-001부터 순번

### 유형별 추가 FR (20%)

#### AI 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 학습 파이프라인 | 데이터 전처리, 학습 실행, 모델 저장 |
| 추론/서빙 | 입력 검증, 모델 로드, 결과 반환 |
| 평가 | 정확도 측정, A/B 비교 |
| 폴백 | 모델 실패 시 대체 동작 |

#### 자동화 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 트리거/스케줄 | 실행 조건, cron 설정 |
| 재시도 | 실패 시 재시도 횟수, 간격 |
| 알림 | 성공/실패 알림 채널 |
| 멱등성 | 중복 실행 시 결과 동일 보장 |

#### BI 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 대시보드 | 차트 유형, 필터, 드릴다운 |
| 리포트 | 생성 주기, 포맷 (PDF/Excel) |
| 지표 정의 | 계산 로직, 집계 규칙 |
| 데이터 새로고침 | 갱신 주기, 수동 새로고침 |

#### 데이터 파이프라인

| 추가 FR 영역 | 예시 |
|---|---|
| 수집 (Ingestion) | 소스 연결, 스키마 매핑 |
| 변환 (Transform) | 정제 규칙, 집계 로직 |
| 적재 (Load) | 대상 저장소, 적재 방식 (upsert/append) |
| 모니터링 | 파이프라인 상태, 에러 알림 |

#### 웹앱 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 인증/인가 | 로그인, 회원가입, 역할 관리 |
| CRUD | 각 엔티티별 생성/조회/수정/삭제 |
| UI/UX | 반응형, 접근성, 다국어 |
| API | 엔드포인트 목록, 요청/응답 포맷 |

#### CLI 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 명령어/서브커맨드 | 명령 목록, 옵션/플래그 |
| 입출력 | stdin/stdout/파일 입출력 포맷 |
| 설정 | 설정 파일 위치, 환경변수 |
| 도움말 | --help 출력, 사용 예시 |

#### 라이브러리 프로젝트

| 추가 FR 영역 | 예시 |
|---|---|
| 공개 API | 함수/클래스 목록, 시그니처 |
| 초기화/설정 | 설정 방법, 기본값 |
| 에러 처리 | 커스텀 예외, 에러 코드 |
| 확장 포인트 | 플러그인, 콜백, 훅 |

## NFR 생성 규칙

1. `references/nfr-checklist.md`의 활성화 판정 규칙에 따라 카테고리 활성/비활성 결정
2. 활성 카테고리: 항목 테이블 + 유형별 조정 사항을 반영하여 NFR 생성
3. 비활성 카테고리: 비활성 사유 템플릿을 채워 "비활성 카테고리" 테이블에 기록
4. 메트릭은 구체적 수치 제시 (사용자 확인/수정 가능)
5. ID 체계: NFR-001부터 순번 (카테고리 접두어 없이 통합 번호)

## 사용자 흐름 생성 규칙

1. **주요 흐름**: 핵심 기능의 happy path를 단계별로 기술
2. **대안 흐름**: 조건 분기 (예: 검색 결과 0건일 때)
3. **예외 흐름**: 에러 상황 처리 (예: API 타임아웃, 인증 실패)
4. 섹션 4(실패 시나리오)가 있으면 해당 내용을 예외 흐름에 반영
5. 유형에 따라 흐름 표현 방식 조정:

| 유형 | 흐름 표현 |
|---|---|
| 웹앱 | 사용자 행위 → 시스템 응답 (UI 중심) |
| CLI | 명령 입력 → 출력 결과 |
| 파이프라인 | 입력 데이터 → 처리 단계 → 출력 |
| AI | 입력 → 전처리 → 추론 → 후처리 → 출력 |
| 자동화 | 트리거 → 실행 → 결과/알림 |
| BI | 데이터 갱신 → 대시보드 조회 → 드릴다운 |
| 라이브러리 | import → 초기화 → API 호출 → 결과 |

---

## 제약사항 보강 규칙

1. 섹션 1 제약사항 테이블을 기반으로 구체화
2. 새로 발견된 제약사항 추가 (기술적 한계, 외부 서비스 제한 등)
3. 근거 컬럼으로 출처 추적
