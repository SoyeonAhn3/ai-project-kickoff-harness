# NFR (Non-Functional Requirements) Checklist

`design-requirements` 스킬이 섹션 9-2 NFR 테이블을 생성할 때 참조하는 체크리스트.
각 카테고리는 활성화 조건에 따라 포함/제외되며, 비활성 시 사유를 기록한다.

---

## 활성화 판정 규칙

킥오프 문서의 아래 필드를 기준으로 판정한다:

| 필드 | 위치 |
|---|---|
| 프로젝트 유형 | 섹션 1 → 기본 정보 → 유형 |
| 프로젝트 목적 | 섹션 1 → 프로젝트 배경 → 프로젝트 목적 |
| 주요 사용자 (내부) | 섹션 1 → 기본 정보 → 주요 사용자 (내부) |
| 기술 스택 | 섹션 1 → 기술 스택 테이블 |

---

## 카테고리 1 — 성능 (Performance)

**활성화 조건**: 항상 활성
**적용 대상**: 모든 프로젝트

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-P01 | 응답 시간 (Response Time) | 주요 기능별 목표 응답 시간 | 높음 |
| NFR-P02 | 처리량 (Throughput) | 단위 시간당 처리 건수 | 중간 |
| NFR-P03 | 리소스 사용량 (Resource Usage) | CPU/메모리 상한 | 낮음 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| 웹앱 | 페이지 로드 시간 (LCP < 2.5s), API 응답 시간 (p95 < 500ms) |
| 파이프라인 | 배치 처리 시간, 레코드당 처리 속도 |
| AI | 추론 지연 시간 (inference latency), 학습 시간 |
| BI | 쿼리 응답 시간, 대시보드 렌더링 시간 |
| CLI | 시작 시간 (< 1s), 명령 실행 시간 |
| 라이브러리 | 함수 호출 오버헤드, 메모리 풋프린트 |
| 자동화 | 트리거→완료 지연 시간, 동시 실행 처리 |

---

## 카테고리 2 — 보안 (Security)

**활성화 조건**: 유형이 웹앱 AND (사용자가 외부 포함 OR 목적이 실무 투입)
**비활성 사유 템플릿**: `보안 카테고리 비활성: [유형]은 외부 사용자 없는 [목적] 프로젝트로, 보안 요구사항 해당 없음`

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-S01 | 인증/인가 (Authentication/Authorization) | 인증 방식, 역할 기반 접근 제어 | 높음 |
| NFR-S02 | 데이터 암호화 (Encryption) | 전송 중/저장 시 암호화 방식 | 높음 |
| NFR-S03 | 입력 검증 (Input Validation) | OWASP Top 10 대응 여부 | 높음 |
| NFR-S04 | 감사 로그 (Audit Logging) | 주요 행위 기록 범위 | 중간 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| 웹앱 | CORS 정책, CSRF 방어, XSS 방지, 세션 관리 |
| AI | 프롬프트 인젝션 방어, 모델 출력 검증 |
| 파이프라인 | 자격 증명 관리 (secrets), 접근 권한 최소화 |

---

## 카테고리 3 — 확장성 (Scalability)

**활성화 조건**: 목적이 실무 투입 OR 사용자가 외부 포함
**비활성 사유 템플릿**: `확장성 카테고리 비활성: [목적] 프로젝트로 확장 요구사항 해당 없음`

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-SC01 | 수평 확장 (Horizontal Scaling) | 인스턴스 추가 시 성능 변화 | 중간 |
| NFR-SC02 | 데이터 증가 대응 (Data Growth) | 데이터 N배 증가 시 성능 영향 | 중간 |
| NFR-SC03 | 동시 사용자 (Concurrent Users) | 목표 동시 접속자 수 | 중간 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| 웹앱 | 동시 접속자 목표, CDN 활용 |
| 파이프라인 | 데이터 볼륨 증가 대응, 파티셔닝 전략 |
| AI | 모델 서빙 스케일링, 배치 추론 |

---

## 카테고리 4 — 가용성 (Availability)

**활성화 조건**: 목적이 실무 투입
**비활성 사유 템플릿**: `가용성 카테고리 비활성: [목적] 프로젝트로 가용성 SLA 불필요`

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-A01 | 목표 가동률 (Uptime SLA) | 99.x% | 높음 |
| NFR-A02 | 장애 복구 시간 (RTO) | 목표 복구 시간 | 중간 |
| NFR-A03 | 데이터 복구 시점 (RPO) | 허용 가능한 데이터 손실 기간 | 중간 |
| NFR-A04 | 모니터링/알림 | 모니터링 범위, 알림 채널 | 중간 |

---

## 카테고리 5 — 유지보수성 (Maintainability)

**활성화 조건**: 항상 활성
**적용 대상**: 모든 프로젝트

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-M01 | 코드 구조 (Code Structure) | 모듈 분리, 관심사 분리 | 중간 |
| NFR-M02 | 테스트 커버리지 (Test Coverage) | 목표 커버리지 % | 중간 |
| NFR-M03 | 문서화 (Documentation) | API 문서, 설치 가이드 | 낮음 |
| NFR-M04 | 로깅 (Logging) | 로그 레벨, 구조화된 로깅 | 중간 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| 라이브러리 | API 문서 필수, 변경 이력 (CHANGELOG), 시맨틱 버저닝 |
| CLI | --help 출력, man page 또는 사용 가이드 |
| 포트폴리오 목적 | README 품질, 코드 가독성 강조 |

---

## 카테고리 6 — 호환성 (Compatibility)

**활성화 조건**: 유형이 웹앱 OR 라이브러리 OR CLI
**비활성 사유 템플릿**: `호환성 카테고리 비활성: [유형]은 멀티 플랫폼 호환 요구사항 없음`

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-C01 | 브라우저/플랫폼 지원 | 지원 대상 목록 | 중간 |
| NFR-C02 | 하위 호환성 (Backward Compatibility) | 기존 버전과의 호환 범위 | 중간 |
| NFR-C03 | 의존성 호환 (Dependency Compat) | 지원 런타임/OS 버전 | 낮음 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| 웹앱 | 브라우저 지원 범위 (Chrome, Firefox, Safari, Edge) |
| 라이브러리 | 런타임 버전 지원 (Python 3.8+, Node 18+ 등) |
| CLI | OS 지원 (Windows, macOS, Linux) |

## 카테고리 7 — 데이터 품질 (Data Quality)

**활성화 조건**: 유형이 파이프라인 OR BI OR AI
**비활성 사유 템플릿**: `데이터 품질 카테고리 비활성: [유형]은 데이터 중심 프로젝트가 아님`

| ID | 항목 | 기본 메트릭 | 우선순위 |
|---|---|---|---|
| NFR-D01 | 정확성 (Accuracy) | 데이터 정확도 기준 | 높음 |
| NFR-D02 | 완전성 (Completeness) | 필수 필드 누락률 | 높음 |
| NFR-D03 | 일관성 (Consistency) | 중복/불일치 허용 범위 | 중간 |
| NFR-D04 | 적시성 (Timeliness) | 데이터 갱신 지연 허용치 | 중간 |

### 유형별 조정

| 유형 | 추가/강조 항목 |
|---|---|
| AI | 학습 데이터 품질, 레이블 정확도, 데이터 편향 점검 |
| BI | 지표 정합성, 집계 규칙 일관성 |
| 파이프라인 | 스키마 검증, 누락 레코드 처리 규칙 |

---

## 사용 방법

1. 킥오프 문서에서 유형/목적/사용자 정보를 읽는다
2. 각 카테고리의 활성화 조건을 판정한다
3. 활성 카테고리: 항목 테이블에서 해당 유형 조정 사항을 반영하여 NFR 테이블 생성
4. 비활성 카테고리: 비활성 사유 템플릿을 채워 섹션 9-2 하단에 기록
5. 사용자에게 메트릭 기본값을 제시하고 확인/수정 요청
