# design-data-model Skill

킥오프 문서(섹션 1~10)를 분석하여 엔티티 정의, 관계, 정합성 규칙, 데이터 생명주기, 마이그레이션 전략을 생성하고 `_kickoff.md`의 섹션 11로 추가한다.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read
  - 2개 이상 발견: 사용자에게 선택 요청
- YAML frontmatter `last_skill`이 `design-architecture`여야 함
  - 아니면 `/design-architecture`를 먼저 실행하도록 안내
- 섹션 10(시스템 아키텍처 상세)이 존재해야 함

---

## STEP 1 — 킥오프 문서 분석

킥오프 문서의 관련 섹션을 Read하여 데이터 모델 도출에 필요한 정보를 추출한다:

| 추출 항목 | 위치 | 용도 |
|---|---|---|
| 프로젝트 유형 | 섹션 1 → 유형 | 유형별 데이터 모델 패턴 선택 |
| 프로젝트 목적 | 섹션 1 → 프로젝트 배경 | 마이그레이션 섹션 활성화 판정 |
| 기존 데이터 구조 | 섹션 3 (있는 경우) | 확장 기반 |
| 기능 요구사항 | 섹션 9-1 | 엔티티/필드 도출 |
| 비기능 요구사항 | 섹션 9-2 | 정합성 규칙, 데이터 품질 기준 |
| 사용자 흐름 | 섹션 9-3 | 데이터 흐름 파악 |
| 시스템 구조 | 섹션 10-1 | 저장소별 엔티티 매핑 |
| 기술 선택 | 섹션 10-2 | DB 유형에 맞는 모델링 |

### 재실행 판정

섹션 11이 이미 존재하는 경우:
- 사용자에게 알림: "섹션 11이 이미 존재합니다. 덮어쓸까요, 아니면 수정할까요?"
- **덮어쓰기**: 기존 섹션 11을 삭제 후 새로 생성
- **수정**: 사용자가 지정한 부분만 Edit

## STEP 2 — 엔티티 도출 및 정의

`references/data-model-template.md`의 엔티티 도출 규칙에 따라 생성한다.

### 2-1. 엔티티 목록 도출

1. 섹션 9-1(FR)의 각 기능에서 관련 데이터 엔티티 추출
2. 섹션 10-1(컴포넌트)에서 저장소 관련 엔티티 추출
3. 섹션 3(v1 데이터 구조)이 있으면 기반으로 확장
4. 유형별 추가 엔티티 반영 (template 참조)

### 2-2. 필드 상세 정의

각 엔티티에 대해:
- 필드명, 타입 (구체적), 제약 조건, 설명
- PK, FK, 인덱스 표시

### 출력 형식

```
### 11-1. 엔티티 정의 (Entity Definition)

#### [엔티티명]

| 필드 | 타입 | 제약 | 설명 |
|---|---|---|---|
| id | INTEGER | PK, AUTO_INCREMENT | 고유 식별자 |
| [필드명] | [타입] | [제약] | [설명] |
```

## STEP 3 — 관계 및 정합성 규칙

### 3-1. 관계 정의

엔티티 간 관계를 정리:

```
### 11-2. 관계 (Relationships)

| 소스 | 대상 | 관계 | 설명 |
|---|---|---|---|
| users | orders | 1:N | 한 사용자가 여러 주문 |
```

### 3-2. 정합성 규칙

섹션 9-2(NFR 데이터 품질) + 비즈니스 규칙에서 도출:

```
### 11-3. 정합성 규칙 (Integrity Rules)

| ID | 규칙 | 대상 | 검증 시점 |
|---|---|---|---|
| IR-001 | [규칙] | [엔티티/필드] | 입력 시 / 배치 |
```

---

## STEP 4 — 데이터 생명주기

섹션 3(v1 볼륨 & 갱신 주기)을 기반으로 확장하거나, 없으면 FR/NFR에서 추정:

```
### 11-4. 데이터 생명주기 (Data Lifecycle)

| 데이터 | 생성 시점 | 갱신 주기 | 보관 기간 | 삭제 정책 |
|---|---|---|---|---|
| [데이터명] | [시점] | [주기] | [기간] | [정책] |
```

---

## STEP 5 — 마이그레이션 전략 (조건부)

### 활성화 조건

| 조건 | 마이그레이션 섹션 |
|---|---|
| 목적 = 실무 투입 | 전체 활성 |
| 유형 = 파이프라인 | 스키마 진화 전략 필수 |
| 목적 = 학습/포트폴리오 | 비활성 또는 간단 초기화만 |

### 비활성 시 출력

```
### 11-5. 마이그레이션/진화 전략

> 마이그레이션 전략은 실무 투입 프로젝트에서 활성화됩니다.
```

### 활성 시 내용

- DB 마이그레이션 도구 선택 (Alembic, Flyway 등)
- 버전 관리 방식
- 롤백 전략
- 스키마 진화 규칙 (파이프라인 유형)

## STEP 6 — 데이터 모델 인터뷰

STEP 2~5 결과를 바탕으로 불명확한 부분에 대해 사용자에게 질문한다.

### 규칙

- **최대 5개 질문**
- 구체적이고 선택지를 포함
- 이미 문서에 답이 있는 질문은 하지 않음

### 질문 예시

| 상황 | 질문 예시 |
|---|---|
| 데이터 보관 기간 미정 | "로그 데이터 보관 기간을 30일/90일/무제한 중 어떻게 설정할까요?" |
| 소프트 삭제 여부 | "삭제된 데이터를 복구 가능하게 유지할까요? (소프트 삭제 vs 하드 삭제)" |
| 캐시 전략 | "자주 조회하는 데이터에 캐싱을 적용할까요? (Redis 등)" |

### 확인 요청

### 8-1. 섹션 11 추가

킥오프 문서의 섹션 10 뒤에 확정된 섹션 11을 추가한다.

- 섹션 11이 없으면: 섹션 10 끝 이후에 추가
- 섹션 11이 있으면 (재실행): 기존 섹션 11을 Edit으로 교체

### 8-2. YAML frontmatter 갱신

```yaml
last_skill: design-data-model
```

### 8-3. Revision History 추가

```
| [오늘 날짜] | 섹션 11 | 데이터 모델 최초 작성 (엔티티 N개, 관계 N개, 정합성 규칙 N개) | design-data-model |
```

### 8-4. 교차 섹션 수정 (해당 시)

데이터 모델링 과정에서 기존 섹션(9, 10)에 수정이 필요한 경우:
1. 사용자에게 수정 여부 확인
2. 승인 시: 해당 섹션 Edit + Revision History에 기록
3. 미승인 시: 섹션 11-3 정합성 규칙에 메모로 기록

## 주의사항

- 모든 출력은 `_kickoff.md` 단일 파일에 추가 (별도 파일 생성 금지)
- 섹션 3(기존 데이터 구조)을 대체하지 않고 공존 — 섹션 11은 확장 버전
- 필드 타입은 선택된 DB 기술에 맞게 구체적으로 명시
- 마이그레이션 전략은 조건부 활성화 — 비활성 사유를 명시
- NoSQL 프로젝트의 경우 엔티티 대신 문서/컬렉션 구조로 표현
- 교차 섹션 수정 시 반드시 사용자 확인 후 진행 + Revision History 기록
