# Data Model Template

`design-data-model` 스킬이 섹션 11(데이터 모델)을 `_kickoff.md`에 작성할 때 참조하는 구조 가이드.
80% 공통 구조 + 20% 프로젝트 유형별 오버라이드로 구성된다.

---

## 섹션 3과의 관계

- **섹션 3 (데이터 구조)**: v1 설계 인터뷰에서 작성한 기초 데이터 구조 (테이블, 볼륨, 갱신 주기)
- **섹션 11 (데이터 모델)**: v2 설계 스킬이 섹션 9(요구사항) + 섹션 10(아키텍처)을 반영하여 확장한 상세 데이터 모델
- 두 섹션은 **공존** — 섹션 11이 섹션 3을 대체하지 않음
- 섹션 3이 없는 경우 (설계 인터뷰 패스): 섹션 11이 데이터 모델의 유일한 출처

## 엔티티 도출 규칙

### 공통 (80%)

1. 섹션 9-1(FR)의 각 기능에서 관련 데이터 엔티티 도출
2. 섹션 10-1(시스템 구조)의 컴포넌트에서 저장소별 엔티티 도출
3. 섹션 3(v1 데이터 구조)이 있으면 기반으로 확장
4. **필드 타입**: 구체적 타입 명시 (VARCHAR(255), INTEGER, TIMESTAMP 등)
5. **제약 조건**: PK, FK, NOT NULL, UNIQUE, DEFAULT, CHECK 등

### 유형별 오버라이드 (20%)

#### 웹앱

| 추가 고려 | 설명 |
|---|---|
| 인증 테이블 | users, sessions, roles, permissions |
| 감사 로그 | created_at, updated_at, created_by 컬럼 |
| 소프트 삭제 | deleted_at 패턴 (해당 시) |
| 인덱스 설계 | 자주 조회하는 컬럼에 인덱스 |

#### 데이터 파이프라인

| 추가 고려 | 설명 |
|---|---|
| 스키마 진화 | 버전 관리, 하위 호환 전략 |
| 파티셔닝 | 날짜/키 기반 파티션 |
| Raw/Processed 분리 | 원본 보존 + 가공 데이터 분리 |
| 메타데이터 | 적재 시간, 소스, 배치 ID |

#### AI 프로젝트

| 추가 고려 | 설명 |
|---|---|
| 학습 데이터 | 피처 테이블, 레이블, 데이터셋 버전 |
| 모델 메타데이터 | 모델 버전, 하이퍼파라미터, 성능 메트릭 |
| 추론 결과 | 입력, 출력, 신뢰도, 타임스탬프 |
| 피드백 데이터 | 사용자 피드백, 정답 레이블 |

#### 자동화

| 추가 고려 | 설명 |
|---|---|
| 작업 이력 | 실행 시간, 상태, 결과, 에러 |
| 설정 데이터 | 트리거 조건, 스케줄, 파라미터 |
| 상태 관리 | 마지막 실행 시점, 체크포인트 |

#### BI

| 추가 고려 | 설명 |
|---|---|
| 차원 테이블 (Dimension) | 시간, 지역, 제품, 사용자 등 |
| 팩트 테이블 (Fact) | 매출, 이벤트, 트랜잭션 등 |
| 집계 테이블 | 사전 집계된 지표 |
| 스냅샷 | 시점별 상태 저장 |

#### CLI

| 추가 고려 | 설명 |
|---|---|
| 설정 파일 | 스키마 (YAML/JSON/TOML) |
| 캐시 데이터 | 로컬 캐시 구조 |
| 출력 포맷 | 구조화된 출력 (JSON/CSV/Table) |

#### 라이브러리

| 추가 고려 | 설명 |
|---|---|
| 공개 타입 | 사용자가 다루는 데이터 타입/인터페이스 |
| 내부 타입 | 라이브러리 내부 상태 |
| 직렬화 | 저장/전송 시 포맷 |

## 관계 표현 규칙

| 관계 유형 | 표기 | 구현 힌트 |
|---|---|---|
| 1:1 | A → B (1:1) | FK + UNIQUE |
| 1:N | A → B (1:N) | B에 FK |
| N:M | A ↔ B (N:M) | 중간 테이블 |
| 자기 참조 | A → A | parent_id FK |

---

## 정합성 규칙 작성 규칙

1. 섹션 9-2(NFR) 데이터 품질 카테고리에서 규칙 도출
2. 비즈니스 규칙 기반 제약 (예: "주문 금액은 0보다 커야 함")
3. 참조 무결성 (FK 관계의 CASCADE/RESTRICT 정책)
4. 검증 시점 명시: 입력 시(동기) / 배치(비동기) / 조회 시(지연)

---

## 데이터 생명주기 작성 규칙

1. 섹션 3(v1 볼륨 & 갱신 주기)이 있으면 기반으로 확장
2. 없으면 FR/NFR에서 추정하여 제시 (사용자 확인)
3. 보관 기간: 법적 요구사항, 스토리지 제약, 비즈니스 필요에 따라 설정
4. 삭제 정책: 소프트 삭제 / 하드 삭제 / 아카이빙

---

## 마이그레이션 전략 활성화 규칙

| 조건 | 마이그레이션 섹션 |
|---|---|
| 목적 = 실무 투입 | 전체 활성 (버전 관리, 롤백 전략) |
| 유형 = 파이프라인 | 스키마 진화 전략 필수 |
| 목적 = 학습/포트폴리오 | 비활성 또는 간단한 초기화 스크립트만 |
