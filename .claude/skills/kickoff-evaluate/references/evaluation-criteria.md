# 평가 기준 상세

## 평가 원칙

1. **건설적 프레임** — 🔴도 "그만둬라"가 아니라 "이렇게 바꾸면 된다"
2. **구체적 근거** — "좋다/나쁘다"가 아니라 "왜 그런지" 설명
3. **킥오프 수준** — 구현 세부사항이 아닌 방향성 평가
4. **대안 필수** — 🟡/🔴 판정 시 반드시 개선 방안 또는 대안 제시

---

## 차원 1 — 차별화

### 핵심 질문

이 프로젝트가 목적에 맞게 눈에 띄는가?

### 목적별 "차별화"의 의미

| 목적 | "차별화"의 의미 | 🟢 기준 | 🟡 기준 | 🔴 기준 |
|---|---|---|---|---|
| 포트폴리오 | 채용 담당자가 "이건 다르다"고 느끼는가? | 명확한 차별 요소 1개 이상 | 유사 프로젝트가 많지만 접근 방식에 고유성 있음 | 이미 범람하는 프로젝트와 구분 불가 |
| 업무 자동화 | 현재 수동 프로세스 대비 명확한 개선이 있는가? | 시간/비용 절감이 측정 가능 | 개선은 있지만 기존 도구로도 가능 | 자동화할 가치가 불분명 |
| 학습 | 새로운 스킬을 실질적으로 습득하는가? | 핵심 학습 목표와 프로젝트가 정렬됨 | 학습 요소는 있지만 깊이가 얕음 | 이미 아는 것만 반복 |
| 실무 투입 | 비즈니스 가치가 측정 가능한가? | KPI/ROI가 명확 | 가치는 있지만 측정 방법 미정 | 비즈니스 임팩트 불명확 |

### context 없을 때

목적을 알 수 없으므로 일반적 기준으로 평가:
- 프로젝트의 고유한 접근 방식이 있는가?
- 단순 튜토리얼 재현이 아닌가?

## 차원 2 — AI 적절성

### 핵심 질문

AI라 부르기 적절한가? 규칙 기반으로 대체 가능한가?

### 적용 조건

프로젝트 유형에 "AI"가 포함된 경우에만 상세 평가. AI 유형이 아니면 "해당 없음"으로 건너뜀.

### 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | ML/DL 모델이 핵심이며 규칙 기반으로 대체 불가능 |
| 🟡 | AI 요소가 있지만 핵심보다 보조 역할, 또는 통계 기법(유사도, 가중합 등)을 AI로 표현 |
| 🔴 | if-else 규칙으로 완전히 대체 가능, AI라 부르기 어려움 |

### 🟡/🔴 시 개선 제안 예시

- "코사인 유사도 + 가중합"은 AI보다 **통계 기반 분석**으로 표현하는 것이 정확
- 진짜 AI 요소를 추가하려면: 임베딩 기반 유사도, 텍스트 분류, 추천 모델 등 고려
- 또는 AI 라벨을 빼고 "데이터 분석 프로젝트"로 리포지셔닝

## 차원 3 — 시장 유효성

### 핵심 질문

6개월 뒤에도 유효한가? 유사 프로젝트가 범람하지 않는가?

### 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | 지속적 수요가 있는 영역, 유사 프로젝트가 적거나 접근 방식이 다름 |
| 🟡 | 수요는 있지만 유사 프로젝트가 많음, 또는 기술 트렌드 변화 가능성 |
| 🔴 | 이미 포화된 영역이거나, 곧 대체될 기술 기반 |

### 점검 포인트

- 이 프로젝트 주제가 GitHub/블로그에서 이미 범람하는가?
- 사용 기술이 6개월 뒤에도 현역인가?
- 타겟 시장/사용자가 실제로 존재하는가?

## 차원 4 — 완성도 기대치

### 핵심 질문

실제 동작하는 데모가 나올 수 있는가?

### 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | MVP 범위가 명확하고, 기술 스택과 제약사항 내에서 실현 가능 |
| 🟡 | 실현 가능하지만 일부 리스크 (외부 의존성, 경험 부족 영역) |
| 🔴 | MVP 범위가 제약사항(기한/인력/기술)에 비해 과대, 또는 핵심 의존성 해결 불가 |

### 점검 포인트

- MVP 기능 수와 시간 예산의 균형
- 외부 서비스 의존도 (API 비용, 접근 제한)
- 핵심 기술에 대한 경험 수준 (context 없으면 판단 불가 → 일반적 기준)
- 데이터 확보 가능성

## 차원 5 — 학습 비용 (조건부)

### 핵심 질문

수준 대비 배울 게 과다하지 않은가? 시간 내 가능한가?

### 활성화 조건

프로젝트 목적이 **포트폴리오** 또는 **학습**인 경우에만 활성화.

### 비활성 사유

업무 자동화/실무 투입은 이미 아는 기술 기반으로 진행하므로 학습량이 병목이 아님. 시간/리소스 부족은 "완성도 기대치" 차원에서 커버됨.

### 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | 새로 배울 기술이 1~2개, 시간 예산 내 학습+구현 가능 |
| 🟡 | 새로 배울 기술이 3개 이상이지만 시간 예산이 충분, 또는 학습 자원 풍부 |
| 🔴 | 기술 수준 대비 학습량이 과다하여 시간 내 완성 불가능 |

### 점검 포인트

- 기술 스택 중 사용자가 처음 접하는 것의 수
- 시간 예산 대비 학습 곡선
- "추천해줘"로 채택된 기술이 몇 개인지 (AI 추천 = 경험 없을 가능성)

### 🟡/🔴 시 개선 제안 예시

- 기술 스택 단순화 (예: Snowflake → SQLite로 시작, 이후 마이그레이션)
- MVP 범위 축소 (핵심 1~2개 기능에 집중)
- 단계별 학습 로드맵 제시

## 차원 6 — 보안 적절성 (조건부)

### 핵심 질문

인증/인가, 민감 데이터, 입력 검증이 고려되었는가?

### 활성화 조건

**웹/앱 프로젝트** + **외부 사용자가 존재**하는 경우에만 활성화.

### 비활성 사유

포트폴리오 + 더미 데이터, 개인 학습용, CLI 도구(로컬 전용)는 외부 노출이 없으므로 보안 점검 불필요.

### 판정 기준

| 판정 | 조건 |
|---|---|
| 🟢 | 인증/인가, 입력 검증, 데이터 보호가 킥오프 문서에 언급됨 |
| 🟡 | 일부 보안 요소 언급되었으나 불완전 (예: 인증만 있고 입력 검증 없음) |
| 🔴 | 외부 사용자가 있는데 보안 관련 언급이 전혀 없음 |

### 점검 항목

- 인증/인가 설계 여부
- 민감 데이터 암호화/마스킹 고려
- 입력 검증 (XSS, SQL injection) 언급
- HTTPS/TLS 적용 계획

## 종합 판정 규칙

| 조건 | 종합 판정 |
|---|---|
| 활성 차원 중 🔴가 1개 이상 | 🔴 재검토 권장 |
| 활성 차원 중 🟡만 있음 (🔴 없음) | 🟡 조건부 적합 |
| 활성 차원 전부 🟢 (또는 해당 없음/비활성) | 🟢 적합 |

- "해당 없음" (AI 적절성 — AI 유형 아닐 때)은 종합 판정에서 제외
- "비활성" (조건부 차원 — 조건 미충족)은 종합 판정에서 제외
