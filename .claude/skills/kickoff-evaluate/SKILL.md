# kickoff-evaluate Skill

킥오프 문서가 완성된 후, "이 프로젝트가 정말 목적에 맞는가?"를 4+2개 차원으로 솔직하게 평가하는 스킬. gap이 "문서 내 모순/누락"을 잡는다면, evaluate는 "프로젝트 자체의 가치와 실현가능성"을 판단한다.

핵심 프레임: **"개발 전에 아는 것이 개발 후에 아는 것보다 낫다."**

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read → YAML frontmatter에서 `last_skill` 확인
  - 2개 이상 발견: frontmatter에 `last_skill` 필드가 있는 파일만 대상, 여러 개면 사용자에게 선택 요청
- YAML frontmatter `last_skill`이 `kickoff-checklist` 이상이어야 함
  - 아니면 `/kickoff-checklist`를 먼저 실행하도록 안내
- 킥오프 문서 `### 프로젝트 배경` 서브섹션이 있으면 목적 기반 평가 활성화, 없으면 일반적 평가만 수행

---

## STEP 1 — 입력 수집

킥오프 문서를 읽어 평가에 필요한 정보를 수집한다.

### 수집 대상

| 출처 | 수집 항목 |
|---|---|
| 킥오프 문서 `### 프로젝트 배경` | 프로젝트 목적, 기술 수준, 시간 예산, 차별화 방향 |
| 킥오프 문서 섹션 1 | 유형, 핵심 기능, 기술 스택, 제약사항 |
| 킥오프 문서 섹션 2~4 | 아키텍처, 데이터 구조, 실패 시나리오 (있는 경우) |
| 킥오프 문서 섹션 5 | gap 점검 결과 |

### 조건부 차원 활성화 판단

수집된 정보로 조건부 차원 활성 여부를 결정한다:

| 조건부 차원 | 활성화 조건 | 판단 기준 |
|---|---|---|
| 학습 비용 | 목적이 포트폴리오 또는 학습 | `### 프로젝트 배경`의 프로젝트 목적 |
| 보안 적절성 | 웹/앱 프로젝트 + 외부 사용자 존재 | 유형 + 사용자 정보 |

- context 없으면 조건부 차원 비활성 (항상 적용 4개만 평가)

---

## STEP 2 — 차원별 평가 수행

`references/evaluation-criteria.md`를 참조하여 활성화된 차원별로 평가한다.

### 평가 차원 (4+2 구조)

**항상 적용 (4개)**:

| # | 차원 | 핵심 질문 |
|---|---|---|
| 1 | 차별화 | 이 프로젝트가 목적에 맞게 눈에 띄는가? |
| 2 | AI 적절성 | AI라 부르기 적절한가? 규칙 기반으로 대체 가능한가? |
| 3 | 시장 유효성 | 6개월 뒤에도 유효한가? 유사 프로젝트가 범람하지 않는가? |
| 4 | 완성도 기대치 | 실제 동작하는 데모가 나올 수 있는가? |

**조건부 적용 (2개)**:

| # | 차원 | 핵심 질문 | 활성화 조건 |
|---|---|---|---|
| 5 | 학습 비용 | 수준 대비 배울 게 과다하지 않은가? | 포트폴리오 또는 학습 목적 시 |
| 6 | 보안 적절성 | 인증/인가, 민감 데이터 처리가 고려되었는가? | 웹/앱 + 외부 사용자 시 |

### AI 적절성 차원 적용 규칙

- 프로젝트 유형에 "AI"가 포함된 경우에만 상세 평가
- AI 유형이 아닌 경우: "해당 없음"으로 표시하고 건너뜀

### 판정 체계

| 판정 | 의미 | 후속 조치 |
|---|---|---|
| 🟢 적합 | 문제 없음 | 없음 |
| 🟡 조건부 | 개선하면 더 좋음 | 구체적 개선 제안 제시 |
| 🔴 재검토 | 방향 자체에 문제 | 대안 제시 + 사용자 결정 요청 |

## STEP 3 — 결과 제시 및 사용자 반응 수집

### 출력 형식

```
## 🪞 정직한 평가

### 종합 판정: [🟢 적합 / 🟡 조건부 적합 / 🔴 재검토 권장]

| # | 평가 차원 | 판정 | 한 줄 피드백 |
|---|---|---|---|
| 1 | 차별화 | 🟢/🟡/🔴 | [피드백] |
| 2 | AI 적절성 | 🟢/🟡/🔴/해당 없음 | [피드백] |
| 3 | 시장 유효성 | 🟢/🟡/🔴 | [피드백] |
| 4 | 완성도 기대치 | 🟢/🟡/🔴 | [피드백] |
| 5 | 학습 비용 | 🟢/🟡/🔴/비활성 | [피드백 또는 "비활성 (업무 자동화/실무 투입 목적)"] |
| 6 | 보안 적절성 | 🟢/🟡/🔴/비활성 | [피드백 또는 "비활성 (외부 사용자 없음)"] |

### 상세 피드백

(🟡/🔴 항목에 대해서만 작성)

#### [차원명] — 🟡 조건부

**현재 상태**: [현재 상황 설명]
**개선 제안**: [구체적 제안]

#### [차원명] — 🔴 재검토

**현재 상태**: [현재 상황 설명]
**문제점**: [구체적 문제]
**대안**: [대안 제시]

---

**어떻게 하시겠습니까?**
1. 현재 방향 유지 (피드백 참고만)
2. 범위 축소/조정 논의
3. 방향 전환 검토
```

### 종합 판정 기준

| 조건 | 종합 판정 |
|---|---|
| 🔴가 1개 이상 | 🔴 재검토 권장 |
| 🟡만 있음 (🔴 없음) | 🟡 조건부 적합 |
| 전부 🟢 (또는 해당 없음/비활성) | 🟢 적합 |

## STEP 4 — (선택) 범위 조정 반영

사용자 선택에 따라 처리한다.

### 사용자 반응별 처리

| 선택 | 처리 |
|---|---|
| 1. 유지 | 평가 결과만 기록, STEP 5로 이동 |
| 2. 조정 | 사용자와 조정 논의 → 킥오프 문서 해당 섹션 Edit으로 수정 → **영향 차원 재평가** → 수정 내용 요약 기록 |
| 3. 전환 | 방향 전환 논의 → 필요 시 `/kickoff-interview`부터 재실행 안내 |

### 조정 후 영향 차원 재평가 (2번 선택 시)

MVP 범위가 변경되면, 영향받는 차원에 대해 STEP 2를 다시 수행한다:

| 조정 유형 | 재평가 대상 차원 |
|---|---|
| 핵심 기능이 v2로 이동 | 차별화, 완성도 기대치 |
| AI 기능이 v2로 이동 | 차별화, AI 적절성 |
| 기술 스택 변경 | 완성도 기대치, 학습 비용(활성 시) |
| 사용자/범위 변경 | 차별화, 시장 유효성 |

**유형 재판별 트리거**: AI 관련 기능이 전부 v2로 밀렸을 때, 킥오프 문서 Section 1 `### 기본 정보`의 부가 유형에서 "AI 프로젝트"를 제거할지 사용자에게 확인한다. 유형이 변경되면 AI 적절성 차원을 "해당 없음"으로 전환한다.

### 유형 변경 시 킥오프 문서 갱신

사용자가 유형 변경을 확인하면, 킥오프 문서를 Edit으로 갱신한다:

1. **`### 기본 정보` 테이블의 "유형" 행**: 변경된 유형으로 교체 + `(평가 후 조정됨)` 표기
2. **변경 전후 안내**: 사용자에게 변경 내용을 명시적으로 보여줌

```
유형이 변경되었습니다:
- 변경 전: [주 유형] + [부가 유형]
- 변경 후: [주 유형] (부가 유형 제거)

킥오프 문서 `### 기본 정보`가 갱신되었습니다.
```

3. **연쇄 영향 확인**: 유형 변경으로 인해 아래 항목도 재확인이 필요할 수 있음을 안내
   - 섹션 5(gap 점검)에서 유형 기반으로 점검한 항목
   - 섹션 6(체크리스트)에서 유형 기반으로 생성한 항목

재평가 결과가 기존 판정과 다를 경우, 변경된 판정으로 갱신하고 사용자에게 변경 사항을 안내한다.

### 조정 시 주의사항

- 킥오프 문서 수정 시 기존 섹션 구조를 유지
- 수정된 부분에 `(평가 후 조정됨)` 표기
- gap 점검 결과(섹션 5)와 충돌하는 수정이 없는지 확인

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-start`를 먼저 실행하도록 안내 |
| 섹션 7 플레이스홀더 없음 | 파일 하단에 섹션 7을 새로 추가 |
| context 없음 | 조건부 차원 비활성, 항상 적용 4개만 평가 + "context 없이 일반 평가만 수행됨" 안내 |
| 사용자가 결과에 강하게 반발 | "이 평가는 참고용이며, 최종 결정은 사용자에게 있습니다" 안내 후 유지로 기록 |
| AI 유형이 아닌데 AI 적절성 질문 | "해당 없음"으로 건너뜀 |

## 주의사항

- 🔴도 **"그만둬라"가 아니라 "이렇게 바꾸면 된다"** — 대안 없는 🔴 판정 금지
- 평가가 사용자를 **낙담시키면 안 됨** — 건설적 피드백 프레임 유지
- context 없이 실행해도 동작 (일반적 평가만, 목적 특화 불가)
- 보안/학습 비용 차원은 조건부 — 해당하지 않는 프로젝트에 강요하지 않음
- 반드시 킥오프 문서를 **Read 먼저** → 이후 Edit으로 추가
- 이미 섹션 7에 내용이 있으면 (재실행 시) 기존 내용을 교체
