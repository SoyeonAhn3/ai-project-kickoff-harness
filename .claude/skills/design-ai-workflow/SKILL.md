# design-ai-workflow Skill

킥오프 문서(섹션 1~11)를 분석하여 AI 입출력 정의, 프롬프트 설계, 모델 선택, 폴백 전략, 평가/모니터링을 생성하고 `_kickoff.md`의 섹션 12로 추가한다. **AI 프로젝트에서만 활성화되며, 비AI 프로젝트는 자동 스킵한다.**

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read
  - 2개 이상 발견: 사용자에게 선택 요청
- YAML frontmatter `completed_skills`에 `design-data-model`이 포함되어야 함
  - 아니면 `/design-data-model`을 먼저 실행하도록 안내

---

## STEP 0 — 활성화 판정

킥오프 문서 섹션 1 → 기본 정보 → 유형을 확인한다.

### 판정 규칙

| 유형 | 판정 | 동작 |
|---|---|---|
| AI 프로젝트 | 활성 | STEP 1부터 진행 |
| AI + 다른 유형 (복합) | 활성 | STEP 1부터 진행 |
| 비AI 프로젝트 | 스킵 | frontmatter 갱신 후 안내 |

### 스킵 시 처리

1. YAML frontmatter `last_skill`을 `design-ai-workflow`로 갱신, `skipped_skills`에 `design-ai-workflow` 추가
2. Revision History에 기록: `| [오늘 날짜] | 섹션 12 | AI 프로젝트가 아니므로 스킵 | design-ai-workflow |`
3. 안내 출력:

```
[design-ai-workflow] 스킵
- 사유: 프로젝트 유형이 [유형]으로, AI 워크플로우 해당 없음
- 다음 단계: /kickoff-gap → /kickoff-checklist
```

## STEP 1 — 킥오프 문서 분석

킥오프 문서의 관련 섹션을 Read하여 AI 워크플로우 설계에 필요한 정보를 추출한다:

| 추출 항목 | 위치 | 용도 |
|---|---|---|
| AI 관련 기능 | 섹션 9-1 (FR) | AI I/O 정의 대상 |
| 성능 NFR | 섹션 9-2 | 지연 시간, 정확도 목표 |
| AI 관련 사용자 흐름 | 섹션 9-3 | 입출력 흐름 파악 |
| AI 컴포넌트 | 섹션 10-1 | 서빙/학습 구조 |
| 모델/프레임워크 | 섹션 10-2 | 기술 선택 기반 |
| AI 데이터 구조 | 섹션 11-1 | 학습/추론 데이터 |

### 재실행 판정

섹션 12가 이미 존재하는 경우:
- 사용자에게 알림: "섹션 12가 이미 존재합니다. 덮어쓸까요, 아니면 수정할까요?"
- **덮어쓰기**: 기존 섹션 12를 삭제 후 새로 생성
- **수정**: 사용자가 지정한 부분만 Edit

## STEP 3 — 프롬프트 설계

`references/ai-workflow-template.md`의 프롬프트 설계 원칙에 따라 각 AI 기능의 프롬프트를 설계한다.

### LLM 기반 프로젝트

각 AI 기능에 대해:
- 목적, 시스템 프롬프트 요약, 사용자 프롬프트 템플릿
- 출력 형식, 예상 토큰, Temperature

### 커스텀 모델 프로젝트

각 AI 기능에 대해:
- 입력 피처, 출력 형식
- 전처리/후처리 파이프라인
- 학습 설정 (에포크, 배치 크기, 학습률 등)

### 출력 형식

```
### 12-2. 프롬프트 설계 (Prompt Design)

#### [프롬프트 1: 기능명]

| 항목 | 내용 |
|---|---|
| 목적 | [작업 설명] |
| 시스템 프롬프트 | [역할/지침 요약] |
| 사용자 프롬프트 템플릿 | [변수 포함 템플릿] |
| 출력 형식 | [JSON/텍스트] |
| 예상 토큰 | 입력: ~N / 출력: ~N |
| 온도 | [0.0~1.0] |
```

## STEP 4 — 모델 선택

### 선택 기준

1. 기능 요구사항 (정확도, 속도, 비용)
2. NFR 제약 (지연 시간, 비용 한도)
3. 섹션 10-2에서 이미 선택된 모델이 있으면 계승

`references/ai-workflow-template.md`의 비용 추정 가이드에 따라 건당/월간 비용을 추정한다.

### 출력 형식

```
### 12-3. 모델 선택 (Model Selection)

| 기능 | 모델 | 선택 이유 | 대안 | 비용 추정 |
|---|---|---|---|---|
| [기능명] | [모델명] | [근거] | [대안] | [비용] |
```

## STEP 7 — AI 워크플로우 인터뷰

STEP 2~6 결과를 바탕으로 불명확한 부분에 대해 사용자에게 질문한다.

### 규칙

- **최대 5개 질문**
- 구체적이고 선택지를 포함

### 질문 예시

| 상황 | 질문 예시 |
|---|---|
| 모델 선택 미정 | "GPT-4o vs Claude Sonnet 중 어떤 모델을 기본으로 사용할까요?" |
| 비용 한도 미정 | "월간 AI API 비용 상한을 얼마로 설정할까요? ($50/$100/$무제한)" |
| 폴백 전략 미정 | "AI 실패 시 기본값 반환 vs 에러 표시 중 어떤 방식을 선호하세요?" |

## STEP 9 — 파일 업데이트

### 9-1. 섹션 12 추가

킥오프 문서의 섹션 11 뒤에 확정된 섹션 12를 추가한다.

- 섹션 12가 없으면: 섹션 11 끝 이후에 추가
- 섹션 12가 있으면 (재실행): 기존 섹션 12를 Edit으로 교체

### 9-2. YAML frontmatter 갱신

```yaml
last_skill: design-ai-workflow
completed_skills:
  - ... (기존 목록)
  - design-ai-workflow
```

### 9-3. Revision History 추가

```
| [오늘 날짜] | 섹션 12 | AI 워크플로우 최초 작성 (AI I/O N개, 프롬프트 N개, 모델 N개) | design-ai-workflow |
```

### 9-4. 교차 섹션 수정 (해당 시)

AI 워크플로우 설계 과정에서 기존 섹션(9~11)에 수정이 필요한 경우:
1. 사용자에게 수정 여부 확인
2. 승인 시: 해당 섹션 Edit + Revision History에 기록
3. 미승인 시: 섹션 12 해당 항목에 메모로 기록

### 완료 출력

```
[design-ai-workflow] 완료
- 대상 파일: [킥오프 문서 경로]
- AI I/O: N개
- 프롬프트: N개
- 모델 선택: N개
- 폴백 시나리오: N개
- 교차 수정: [있음/없음]
- 다음 단계: /kickoff-gap → /kickoff-checklist
```

## 주의사항

- 모든 출력은 `_kickoff.md` 단일 파일에 추가 (별도 파일 생성 금지)
- **비AI 프로젝트는 STEP 0에서 자동 스킵** — 사용자에게 불필요한 질문 없이 처리
- 프롬프트 설계 시 프롬프트 인젝션 방어를 반드시 고려
- 비용 추정은 현실적 수치 제시 (과소/과대 추정 금지)
- 모델명은 최신 모델 기준으로 제안 (사용자 확인)
- 교차 섹션 수정 시 반드시 사용자 확인 후 진행 + Revision History 기록
