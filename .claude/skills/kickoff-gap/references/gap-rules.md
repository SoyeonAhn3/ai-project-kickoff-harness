# 누락/모순 점검 규칙

## 점검 원칙

1. **명확한 모순만** 잡는다 — 강한 확신이 있는 것만 지적
2. **애매한 건** 사용자에게 질문 — AI가 임의로 판단하지 않음
3. **오탐 최소화** — 없는 문제를 만들어내는 것보다 놓치는 게 낫다
4. **킥오프 수준** — 구현 레벨 세부사항은 점검하지 않음

---

## 점검 카테고리

### 카테고리 1 — 기술 선택 불일치

프로필에 명시된 기술 스택이 서로 충돌하거나, 제약사항과 맞지 않는 경우.

**점검 항목:**
- 기술 스택과 제약사항 간 충돌 (예: "Docker 불가"인데 컨테이너 기반 아키텍처)
- 프론트엔드/백엔드 기술 간 호환성 문제
- 선택한 도구가 데이터 볼륨을 감당 불가 (예: SQLite에 초당 1만 건)

**판정 기준:**
- 명확한 기술적 비호환 → **모순**으로 지적
- "보통 이 조합은 잘 안 씀" 수준 → 지적하지 않음

---

### 카테고리 2 — 데이터 흐름 누락

파이프라인 흐름에서 빠진 단계가 있는 경우.

**점검 항목:**
- 데이터 소스는 정의되었지만 수집 방법이 없음
- 처리 결과는 있지만 저장 위치가 없음
- 입력 데이터의 형식 변환이 필요한데 명시되지 않음
- 외부 서비스 호출은 있지만 인증/키 관리 언급 없음
- **배포 환경 접근 제한 리스크**: 외부 데이터 수집 방식이 비공식 API 또는 웹 스크래핑에 의존하는 경우, 클라우드/데이터센터 IP에서 차단될 가능성이 있음. 아래 패턴에 해당하면 경고:
  - 공식 API가 아닌 웹페이지 스크래핑 라이브러리 사용 (예: yfinance `stock.info`, BeautifulSoup 크롤링 등)
  - 공식 API가 종료/폐지된 서비스에 비공식 방식으로 접근
  - 외부 서비스가 봇/자동화 접근을 제한한다고 알려진 경우
- **Fallback 미정의**: 위 리스크가 존재하는데 대체 데이터 소스나 Fallback 방식이 정의되지 않은 경우

**판정 기준:**
- 흐름 상 반드시 필요한 단계 누락 → **누락**으로 지적
- 비공식 API/스크래핑 의존 + Fallback 미정의 → **누락**으로 지적 (배포 시 서비스 불가 위험)
- "있으면 좋은" 수준 → 지적하지 않음

---

### 카테고리 3 — 시간/스케줄 충돌

실행 주기, 갱신 시간, 의존 관계 사이에 타이밍 문제가 있는 경우.

**점검 항목:**
- 스케줄러 실행 시간과 데이터 갱신 시간 충돌 (예: 데이터가 09시에 갱신되는데 배치가 08시 실행)
- 의존 데이터가 준비되기 전에 후속 처리가 시작
- 실시간 요구사항인데 배치 처리 설계

**판정 기준:**
- 명시적 시간 충돌 → **모순**으로 지적
- 시간이 명시되지 않은 경우 → **애매** → 사용자에게 질문

---

### 카테고리 4 — 실패 처리 누락

시스템이 실패할 수 있는 상황에 대한 대응이 정의되지 않은 경우.

**점검 항목:**
- AI 결과가 틀렸을 때의 폴백이 없음
- 외부 서비스 장애 시 동작이 정의되지 않음
- 데이터 중복 방지 방식 누락
- 알림/모니터링 대상이 정의되지 않음

**판정 기준:**
- 핵심 흐름에서 실패 시 시스템이 멈추는 경우 → **누락**으로 지적
- 부가 기능의 실패 처리 → 지적하지 않음 (킥오프 수준)

---

### 카테고리 5 — MVP 범위 모순 및 교차 일관성

MVP에 포함된 기능과 다른 항목 간 불일치. 핵심 기능 ↔ 외부 의존성 ↔ 완료 조건 간 교차 검증 포함.

**점검 항목:**
- MVP에 포함된 기능이 "이후 확장"으로 분류된 의존성을 필요로 함
- MVP 범위가 제약사항(기한, 인력)에 비해 과다
- 핵심 기능이 MVP에서 빠져있음

**교차 일관성 점검 (양방향):**
- **기능 → 의존성**: MVP 기능에 필요한 외부 서비스/API가 의존성 표에서 "선택" 또는 "v2"로 표시되어 있으면 충돌
- **의존성 → 기능**: 의존성 표에서 "필수"로 분류된 항목이 핵심 기능 표에서 "선택"으로 되어 있으면 충돌
- **기능 → 완료 조건**: MVP 기능이 완료 조건에 반영되지 않았으면 누락
- **완료 조건 → 기능**: 완료 조건에 포함된 항목이 핵심 기능에서 "선택" 또는 MVP 제외로 되어 있으면 충돌

**판정 기준:**
- 기능 의존성 충돌 → **모순**으로 지적
- 교차 일관성 불일치 → **모순**으로 지적 + 사용자에게 "MVP 필수로 유지할지, 선택으로 내릴지" 결정 요청
- 범위 과다 여부 → **애매** → 사용자에게 "이 범위가 [기한] 내에 가능한가요?" 질문

---

### 카테고리 6 — 평가 지적 항목 재확인

evaluate 스킬에서 🟡/🔴로 지적된 항목이 킥오프 문서 내에서 일관되게 반영되었는지 확인.

**점검 항목:**
- evaluate에서 범위 조정이 이루어진 경우, 킥오프 문서의 해당 섹션이 수정되었는지
- 조정 결과와 기존 섹션(기술 스택, MVP 범위 등) 간 새로운 모순이 없는지
- evaluate를 실행하지 않은 경우 이 카테고리는 건너뜀

**판정 기준:**
- 조정 내용과 기존 섹션 간 충돌 → **모순**으로 지적
- evaluate 미실행 → 점검 불가 (건너뜀)

---

### 카테고리 7 — v2 설계 섹션 교차 정합성

v2 설계 스킬(섹션 9~12)이 작성된 경우, 섹션 간 교차 정합성을 검증한다. **v2 설계 스킬 미실행 시 이 카테고리는 건너뜀.**

**점검 항목:**

#### 7-1. NFR vs 제약사항 산술 정합성
- NFR 메트릭(섹션 9-2)에 언급된 수치가 제약사항(섹션 1)과 모순되지 않는지
- 예: NFR에 "전체 수집 30분 이내"인데 제약사항에 "1시간 배치"로 기술

#### 7-2. 사용자 흐름 vs 아키텍처 진입점
- 사용자 흐름(섹션 9-3)의 시작점/실행 방식이 아키텍처(섹션 10-1)와 일치하는지
- 예: 사용자 흐름에 `python src/main.py`인데 아키텍처에 진입점이 `collect.py`

#### 7-3. 아키텍처 vs 데이터 모델 정합성
- 아키텍처 컴포넌트(섹션 10-1)의 통신/저장 방식이 데이터 모델(섹션 11)과 일치하는지
- ADR(섹션 10-5)의 결정 사항이 데이터 생명주기(섹션 11-4)와 모순되지 않는지
- 예: ADR에 "로그는 콘솔 출력"인데 데이터 모델에 "로그 테이블 INSERT"

#### 7-4. v1 섹션 vs v2 섹션 일관성
- v1 핵심 기능(섹션 1)과 v2 FR(섹션 9-1)의 범위가 일치하는지
- v1 실패 시나리오(섹션 4)와 v2 폴백/에러 처리(섹션 12-4)가 모순되지 않는지
- v1 데이터 구조(섹션 3)와 v2 데이터 모델(섹션 11)이 호환되는지

**판정 기준:**
- 명확한 수치/명칭 불일치 → **모순**으로 지적
- v1/v2 범위 차이 → **모순**으로 지적 + 어느 쪽을 기준으로 할지 결정 요청
- v2 설계 스킬 미실행 → 점검 불가 (건너뜀)

## 점검 범위 조건

### 설계 인터뷰 패스 시 (섹션 2~4 없음)

- 카테고리 2, 3 → 점검 불가 (데이터 없음)
- 카테고리 1 → 제약사항과 기술 스택 간 충돌만 점검
- 카테고리 4 → 프로필에서 언급된 범위만 점검
- 카테고리 5 → 정상 점검

### v2 설계 미수행 시 (섹션 9~12 없음)

- 카테고리 7 → 건너뜀 (v2 설계 섹션 없음)
- 카테고리 1~6 → 정상 점검

## 결과 기록 형식

### 모순/누락 발견 시

```markdown
| # | 유형 | 내용 | 상태 |
|---|---|---|---|
| 1 | 모순 | [구체적 내용] | 🔴 미해결 / ✅ 해결 |
| 2 | 누락 | [구체적 내용] | 🔴 미해결 / ✅ 해결 |
```

### 해결된 항목 기록

```markdown
| # | 유형 | 내용 | 상태 | 해결 방법 |
|---|---|---|---|---|
| 1 | 모순 | [내용] | ✅ 해결 | [사용자 결정 내용] |
```

### 점검 결과 없음

```markdown
점검 결과: 명확한 누락/모순이 발견되지 않았습니다.
```
