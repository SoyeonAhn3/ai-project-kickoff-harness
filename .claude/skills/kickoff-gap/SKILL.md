# kickoff-gap Skill

project_kickoff.md를 분석하여 누락된 결정사항이나 모순되는 내용을 발견하고, 섹션 5로 결과를 추가하는 스킬. 이 하네스의 핵심 차별점 — "개발 전에 문제를 찾아내는 도구".

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read → YAML frontmatter에서 `last_skill` 확인
  - 2개 이상 발견: frontmatter에 `last_skill` 필드가 있는 파일만 대상, 여러 개면 사용자에게 선택 요청
- YAML frontmatter `last_skill`이 `kickoff-profile` 이상이어야 함
  - 아니면 `/kickoff-profile`을 먼저 실행하도록 안내

---

## STEP 1 — 문서 읽기

킥오프 문서(`{프로젝트명}_kickoff.md`)를 Read하여 전체 내용을 파악한다.

### 확인 사항

- 파일 존재 여부
- 어떤 섹션이 포함되어 있는지 (설계 패스 여부 판단)
- 섹션 5 플레이스홀더 위치 확인

---

## STEP 2 — 점검 수행

`references/gap-rules.md`의 5개 카테고리를 순서대로 점검한다.

### 점검 카테고리

| # | 카테고리 | 점검 대상 |
|---|---|---|
| 1 | 기술 선택 불일치 | 기술 스택 vs 제약사항, 기술 간 호환성 |
| 2 | 데이터 흐름 누락 | 파이프라인 빈 단계, 인증/키 관리 |
| 3 | 시간/스케줄 충돌 | 실행 시간 vs 갱신 시간, 의존 관계 |
| 4 | 실패 처리 누락 | 폴백, 중복 방지, 모니터링 |
| 5 | MVP 범위 모순 | 기능 의존성, 범위 vs 제약 |
| 6 | 평가 지적 항목 재확인 | evaluate 조정 결과와 기존 섹션 간 충돌 (evaluate 미실행 시 건너뜀) |

### 설계 패스 시

- 카테고리 2, 3 → 건너뜀
- 카테고리 1, 4 → 프로필 범위만 점검
- 카테고리 5 → 정상 점검

---

## STEP 3 — 결과 제시 및 해결

### 발견 사항이 있는 경우

```
## 🔍 누락/모순 점검 결과

project_kickoff.md를 분석한 결과 [N]건의 이슈를 발견했습니다.

### 명확한 이슈 (해결 필요)

| # | 유형 | 내용 |
|---|---|---|
| 1 | 모순 | [구체적 설명] |
| 2 | 누락 | [구체적 설명] |

### 확인이 필요한 사항

| # | 내용 | 질문 |
|---|---|---|
| A | [상황 설명] | [사용자에게 물어볼 질문] |

---

명확한 이슈에 대해 어떻게 해결할지 말씀해주세요.
확인 사항은 답변해주시면 반영합니다.
```

### 발견 사항이 없는 경우

```
## 🔍 누락/모순 점검 결과

project_kickoff.md를 분석한 결과, 명확한 누락이나 모순이 발견되지 않았습니다.
기획이 일관성 있게 정리되어 있습니다.

다음 단계로 `/kickoff-checklist`를 실행하면 개발 착수 체크리스트가 생성됩니다.
```

### 사용자 응답 처리

- 명확한 이슈: 사용자가 해결 방법을 제시하면 "문제 + 해결" 형태로 기록
- 확인 사항: 사용자 답변을 반영하여 상태 업데이트
- 필요 시 project_kickoff.md의 해당 섹션도 수정

---

## STEP 4 — 파일 업데이트

모든 이슈가 해결(또는 확인)된 후:

1. project_kickoff.md의 섹션 5 플레이스홀더를 실제 내용으로 교체
2. Edit으로 `## 5. 누락/모순 점검 결과` 아래에 결과 테이블 작성

### 섹션 5 작성 형식

```markdown
## 5. 누락/모순 점검 결과

| # | 유형 | 내용 | 상태 | 해결 방법 |
|---|---|---|---|---|
| 1 | 모순 | [내용] | ✅ 해결 | [해결 내용] |
| 2 | 누락 | [내용] | ✅ 해결 | [해결 내용] |

점검 일시: [날짜]
```

발견 사항이 없는 경우:

```markdown
## 5. 누락/모순 점검 결과

점검 결과: 명확한 누락/모순이 발견되지 않았습니다.

점검 일시: [날짜]
```

---

## STEP 5 — 완료 및 안내

### YAML frontmatter 갱신

킥오프 문서의 YAML frontmatter `last_skill`을 `kickoff-gap`으로 변경 (Edit 사용)

### 안내 출력

```
## ✅ 누락/모순 점검 완료

**발견**: [N]건 (모순 [n]건, 누락 [m]건)
**해결**: 전체 해결됨

`{프로젝트명}_kickoff.md` 섹션 5가 업데이트되었습니다.

다음 단계로 `/kickoff-checklist`를 실행하면 개발 착수 체크리스트가 생성됩니다.
```

---

## 출력 형식

이 스킬은 **킥오프 문서의 해당 섹션을 채운다**:
- 대상: `{프로젝트명}_kickoff.md` 섹션 5
- 방식: 섹션 5 플레이스홀더 교체 (Edit 사용)
- YAML frontmatter `last_skill` 갱신

---

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-profile`을 먼저 실행하도록 안내 |
| 섹션 5 플레이스홀더 없음 | 파일 하단에 섹션 5를 새로 추가 |
| 사용자가 이슈 해결을 거부 | "미해결" 상태로 기록, 개발 시 주의 필요 메모 |
| 점검 중 새로운 정보 발견 | 사용자에게 공유 후 필요 시 해당 섹션 업데이트 |

---

## 주의사항

- 반드시 킥오프 문서를 **Read 먼저** → 이후 Edit으로 추가
- **명확한 것만 지적** — 오탐은 사용자 신뢰를 떨어뜨림
- 애매한 건 반드시 **사용자에게 질문** 후 기록
- 설계 패스 시 점검 범위가 좁아지는 것은 정상 — 무리하게 추측하지 않음
- 이미 섹션 5에 내용이 있으면 (재실행 시) 기존 내용을 유지하고 추가 발견만 append
