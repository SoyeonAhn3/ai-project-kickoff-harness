# kickoff-profile Skill

킥오프 문서가 이전 스킬들(start → interview → suggest)에 의해 채워진 상태에서, 전체 내용의 일관성과 품질을 검토하고 사용자 확인을 받는 최종 검토 스킬. 이 파일은 이후 모든 단계의 Single Source of Truth.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read → YAML frontmatter 확인
  - 2개 이상 발견: frontmatter에 `completed_skills` 필드가 있는 파일만 대상, 여러 개면 사용자에게 선택 요청
- YAML frontmatter `completed_skills`에 `kickoff-suggest`가 포함되어야 함
  - 아니면 `/kickoff-suggest`를 먼저 실행하도록 안내

---

## STEP 1 — 킥오프 문서 전체 읽기 및 완성도 확인

킥오프 문서를 Read하여 각 섹션의 완성도를 확인한다.

### 점검 항목

| 점검 대상 | 확인 내용 |
|---|---|
| Section 1 `### 기본 정보` | 모든 항목이 "미정"이 아닌 실제 값으로 채워져 있는지 |
| Section 1 `### 핵심 기능` | 플레이스홀더가 아닌 기능 테이블이 있는지 |
| Section 1 `### 채택된 AI 제안` | 채택 테이블 또는 "전부 패스" 표기가 있는지 |
| Section 1 `### 기술 스택` | 플레이스홀더가 아닌 기술 테이블이 있는지 |
| Section 1 `### 제약사항` | 플레이스홀더가 아닌 제약사항 테이블이 있는지 |
| Sections 2-4 | 설계 데이터로 채워져 있거나, 패스 시 생략되어 있는지 |
| Sections 5-8 | 플레이스홀더가 유지되어 있는지 (이후 스킬이 채울 예정) |

---

## STEP 2 — 품질 검토

### 검토 항목

1. **일관성**: 기술 스택에 언급된 기술이 파이프라인 흐름(Section 2)과 일치하는지
2. **AI 추천 표기**: "추천해줘"로 답한 항목에 `(AI 추천)` 표기가 있는지
3. **빈 항목 처리**: 정보가 없는 항목이 빈칸이 아닌 "미정"으로 표기되어 있는지
4. **헤더 포맷**: 섹션 헤더(`## 1.`, `## 2.` 등)가 이후 스킬 파싱에 적합한 형식인지
5. **한 줄 설명**: `# Project Kickoff` 아래 `>` 인용 설명이 프로젝트를 잘 요약하는지
6. **사용자 표기 분리**: 포트폴리오 목적 프로젝트에서 "주요 사용자 (외부 표기)" 행이 존재하고, 외부 노출에 적절한 톤인지 확인. "본인 (학습용)" 같은 내부 표현이 외부 표기에 그대로 들어가 있으면 사용자에게 수정 제안

### 문제 발견 시

- 자동 수정 가능한 문제 (빈칸 → "미정", AI 추천 표기 누락): Edit으로 즉시 수정
- 내용 판단이 필요한 문제 (일관성 불일치, 정보 부족): 사용자에게 질문

## STEP 3 — 사용자 확인

검토 결과를 요약하여 사용자에게 확인을 받는다.

### 출력 형식

```
## 📄 킥오프 문서 최종 검토

**파일**: `{프로젝트명}_kickoff.md`

### 검토 결과:

| 섹션 | 상태 |
|---|---|
| 1. 프로젝트 프로필 | ✅ 완성 |
| 2. 시스템 아키텍처 | ✅ 완성 / ⏭️ 생략 (설계 패스) |
| 3. 데이터 구조 | ✅ 완성 / ⏭️ 생략 (설계 패스) |
| 4. 실패 시나리오 | ✅ 완성 / ⏭️ 생략 (설계 패스) |
| 5. 누락/모순 점검 | ⏳ 대기 (/kickoff-gap에서 추가) |
| 6. 개발 착수 체크리스트 | ⏳ 대기 (/kickoff-checklist에서 추가) |
| 7. 정직한 평가 | ⏳ 대기 (/kickoff-evaluate에서 추가) |
| 8. 완료 조건 | ⏳ 대기 (/kickoff-done에서 추가) |

[자동 수정 사항 요약 (있는 경우)]

수정할 부분이 있으면 말씀해주세요.
없으면 `/kickoff-gap`을 실행하여 누락/모순을 점검합니다.
```

### 사용자 수정 요청 시

- 해당 부분을 수정하고 파일을 다시 저장
- 수정 완료 후 다시 확인 요청

---

## STEP 4 — 완료 및 안내

사용자가 확인하면:

### YAML frontmatter 갱신

킥오프 문서의 YAML frontmatter `last_skill`을 `kickoff-profile`로 변경, `completed_skills`에 `kickoff-profile` 추가 (Edit 사용)

### 안내 출력

```
## ✅ 킥오프 문서 최종 검토 완료

**파일**: `{프로젝트명}_kickoff.md`

다음 단계로 `/kickoff-evaluate`를 실행하면 프로젝트의 가치와 실현가능성을 평가합니다.
```

---

## 출력 형식

이 스킬은 **기존 킥오프 문서를 검토하고 품질을 정리**한다:
- 대상: `{프로젝트명}_kickoff.md`
- 방식: 자동 수정 가능한 항목은 Edit, 내용 판단은 사용자에게 질문
- YAML frontmatter `last_skill` 및 `completed_skills` 갱신
- 이후 `kickoff-gap`과 `kickoff-checklist`가 킥오프 문서를 읽고 섹션 5, 6을 추가

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-start`를 먼저 실행하도록 안내 |
| 인터뷰 데이터 미완성 (핵심 기능이 플레이스홀더) | `/kickoff-interview`를 먼저 실행하도록 안내 |
| 사용자가 대폭 수정 요청 | 수정 반영 후 파일 재저장 + 재확인 |

---

## 주의사항

- 섹션 헤더 포맷(## 1. / ## 2. 등)을 정확히 유지 — 이후 스킬이 파싱에 의존
- 섹션 5-8 플레이스홀더를 삭제하지 않음 — 이후 스킬이 이 위치에 내용 추가
- 이 스킬은 문서를 새로 생성하지 않음 — kickoff-start가 생성한 문서를 검토/정리만 수행
- AI가 수정한 내용은 사용자 확인 후에만 확정
