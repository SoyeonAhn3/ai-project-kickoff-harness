# project_kickoff.md 템플릿

`kickoff-start`가 킥오프 문서를 생성하고, 각 스킬이 해당 섹션을 채울 때 참조하는 표준 구조다.
각 섹션의 헤더와 항목은 이후 스킬(kickoff-gap, kickoff-checklist 등)이 파싱하므로 **헤더 포맷을 변경하지 말 것**.

---

## 전체 구조

```markdown
---
project: [프로젝트명]
created: [날짜]
language: [ko/en]
last_skill: kickoff-start
completed_skills:
  - kickoff-start
skipped_skills: []
---

# Project Kickoff — [프로젝트명]

### 핵심 기능

> 이 섹션은 `/kickoff-interview` 실행 시 작성됩니다.

| # | 기능 | MVP 포함 |
|---|---|---|
| 1 | [P5 답변에서 나열] | ✅ / ⬜ 선택 |
| 2 | ... | ... |

**행 분리 규칙**: 하나의 기능에 필수/선택 구성 요소가 섞여 있으면 행을 분리한다. 외부 서비스 의존성 표의 필수/선택 분류와 대응되어야 한다.

```
변경 전: | 1 | API 데이터 수집 (A, B, C, D) | ✅ |
변경 후: | 1 | API 데이터 수집 — 핵심 (A, B)    | ✅     |
         | 2 | API 데이터 수집 — 추가 (C, D)    | ⬜ 선택 |

## 작성 규칙

1. **프로젝트명**: 사용자가 명시하지 않으면 AI가 아이디어에서 적절한 이름 생성
2. **AI 추천 표기**: 인터뷰에서 "추천해줘"로 답한 항목은 값 뒤에 `(AI 추천)` 표기
3. **빈 항목 처리**: 정보가 없는 항목은 "미정" 표기 (빈칸으로 두지 않음)
4. **플레이스홀더**: 아직 작성되지 않은 섹션은 `> 이 섹션은 /스킬명 실행 시 작성됩니다.` 문구만 유지 (이후 스킬이 내용 추가)
5. **설계 패스 시**: 섹션 2, 3, 4를 통째로 생략 (헤더도 없음)
6. **언어**: YAML frontmatter `language` 필드에 따라 작성 (`ko` = 한국어, `en` = English). 모든 스킬의 질문, 섹션 내용, 완료 메시지를 해당 언어로 출력한다. `language` 필드가 없으면 한국어 기본.
7. **YAML frontmatter**: 각 스킬 완료 시 `last_skill` 필드를 해당 스킬명으로 갱신하고, `completed_skills`에 추가. 스킵된 스킬은 `skipped_skills`에 추가
8. **문서 생성 시점**: `kickoff-start`가 초기 문서를 생성하고, 이후 스킬이 각 섹션을 채움
9. **Revision History**: 설계 스킬(섹션 9~12)이 기존 섹션을 수정할 때 날짜, 대상 섹션, 변경 내용, 스킬명을 기록
