# kickoff-interview Skill

프로젝트 유형에 맞는 기획/설계 질문을 통해 요구사항을 빠짐없이 수집하는 인터뷰 스킬.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read → YAML frontmatter에서 `last_skill` 확인
  - 2개 이상 발견: frontmatter에 `last_skill` 필드가 있는 파일만 대상, 여러 개면 사용자에게 선택 요청
- YAML frontmatter `last_skill`이 `kickoff-start` 이상이어야 함
- 킥오프 문서 Section 1 `### 기본 정보`에서 프로젝트 유형 확인, `### 프로젝트 배경`에서 context 데이터 확인 (있는 경우)

---

## 공통 인터뷰 규칙

모든 STEP에 적용되는 규칙:

1. **질문 방식**: B+C (묶음 3~8개 제시 + 부족하면 추가 1~2개)
2. **"잘 모르겠어" / "추천해줘"**: 정식 답변으로 인정. 즉시 AI 추천 제시
3. **AI 추천 우선순위**: 목적에 맞는 것 > 쉬운 것
4. **사용자 답변 우선**: 사용자가 아는 게 있으면 본인 답변 우선
5. **사소한 디테일 금지**: 킥오프 수준 유지, 구현 세부사항은 묻지 않음
6. **언어**: 사용자가 사용하는 언어로 진행

---

## 컨텍스트 활용 규칙

킥오프 문서 `### 프로젝트 배경`(사용자 컨텍스트)이 존재하면 기획 인터뷰에 목적별 추가 질문을 삽입한다.

| 목적 | 추가 질문 (기획 질문 마지막에 삽입) |
|---|---|
| 포트폴리오 | "이 프로젝트에서 가장 보여주고 싶은 기술적 포인트는?" |
| 업무 자동화 | "현재 수동 프로세스의 가장 큰 병목은?" |
| 학습 | "이전에 시도해봤지만 실패한 경험이 있나요?" |
| 실무 투입 | "SLA나 가용성 요구사항이 있나요?" |

- 기술 수준이 "초급"이면 설계 인터뷰에서 "추천해줘" 선택지를 더 적극적으로 제시
- 컨텍스트 없으면 추가 질문 없이 기존 동작 유지

---

## STEP 1 — 기획 인터뷰 (What)

"무엇을 만들 것인가"에 집중. 비기술적 사용자도 답변 가능한 수준.

### 동작

1. `references/planning-questions.md` 참조
2. 공통 질문 7개 + 유형별 추가 질문 3개를 한 묶음으로 제시
3. 컨텍스트가 있으면 목적별 추가 질문 1개를 묶음 끝에 삽입
4. 복합 유형인 경우: 주요 유형 3개 + 부가 유형 1개

### 출력 형식

```
## 📝 기획 인터뷰를 시작합니다

프로젝트 유형 **[유형명]**에 맞춰 질문드리겠습니다.
모르는 건 "잘 모르겠어" 또는 "추천해줘"라고 답하셔도 됩니다.

---

**1.** [질문 1]
**2.** [질문 2]
**3.** [질문 3]
...

---

편하게 번호별로 답변해주세요. 한꺼번에 답하셔도 되고, 하나씩 답하셔도 됩니다.
```

### 답변 처리

- 각 답변을 내부적으로 정리 (수집 목적에 매핑)
- "잘 모르겠어" → 해당 항목에 AI 추천을 즉시 제시하고 채택 여부 확인
- 정보 부족 시 → 해당 영역에서 추가 질문 1~2개 (C방식)

### P1-b — 외부 표기용 사용자 질문

P1(주요 사용자) 답변이 "본인", "학습용", "테스트용" 등 내부적 표현이고, 킥오프 문서 `### 프로젝트 배경`에서 목적이 포트폴리오인 경우, 추가 질문을 한다:

```
이 프로젝트를 GitHub이나 이력서에 소개할 때, 사용자를 어떻게 표현하면 좋을까요?
예: "금융 데이터 분석이 필요한 투자자/리서처", "생산 현장의 품질 관리자"

잘 모르겠으면 "추천해줘"라고 해주세요.
```

- 포트폴리오 목적이 아니면 이 질문은 건너뜀
- P1 답변이 이미 외부 노출에 적합한 표현이면 건너뜀 (예: "금융 분석가")

### 기획 인터뷰 완료 확인

모든 답변 수집 후:

```
## ✅ 기획 인터뷰 완료

수집된 내용을 정리합니다:

| 항목 | 내용 |
|---|---|
| 사용자 (내부) | [P1 답변 요약] |
| 사용자 (외부 표기) | [P1-b 답변 또는 P1과 동일] |
| 핵심 문제 | [답변 요약] |
| 데이터 소스 | [답변 요약] |
| 결과 형태 | [답변 요약] |
| MVP 범위 | [답변 요약] |
| [유형별 항목] | [답변 요약] |

수정할 부분이 있으면 말씀해주세요. 없으면 설계 인터뷰로 넘어갑니다.
```

## STEP 2 — 설계 인터뷰 진입 확인

기획 인터뷰 완료 후, 설계 인터뷰 진행 여부를 확인한다.

### 출력 형식

```
## 🔧 설계 인터뷰

기술적인 구조(아키텍처, 데이터 구조, 실패 시나리오)를 구체화하는 단계입니다.

- 진행하면: 더 완성도 높은 kickoff 문서가 만들어집니다
- 패스하면: 기획 정보만으로 진행합니다 (나중에 다시 할 수 있음)

**진행하시겠습니까?** (진행 / 패스)
```

### 분기

- **"패스"** → STEP 4로 이동 (완료 안내)
- **"진행"** → STEP 3으로 이동

---

## STEP 3 — 설계 인터뷰 (How)

"어떻게 만들 것인가"에 집중. 기획 답변을 기반으로 기술 구조를 구체화한다.

### 중복 제거 규칙

각 영역의 질문을 제시하기 전에, 기획 인터뷰(STEP 1) 답변에서 해당 영역의 내용이 이미 나왔는지 확인한다.

| 판단 | 기준 | 동작 |
|---|---|---|
| 충분 | 기획 답변에서 해당 영역의 핵심 정보가 구체적으로 나옴 | 기획 답변을 요약 제시 → "이 내용이 맞나요?" 확인만 → 맞으면 건너뜀 |
| 부분 | 방향은 나왔지만 세부사항 부족 | 이미 나온 내용을 언급하고, 부족한 부분만 추가 질문 |
| 없음 | 해당 영역 관련 언급 없음 | 전체 질문 제시 |

#### 영역별 기획 답변 대응표

| 설계 영역 | 겹칠 수 있는 기획 답변 |
|---|---|
| 영역 1 (파이프라인 흐름) | P3 데이터 소스, P4 결과 형태, P5 핵심 기능, P6 유형별 질문 |
| 영역 2 (데이터 구조) | P3 데이터 소스, P5 핵심 기능 |
| 영역 3 (볼륨/갱신 주기) | P3 데이터 소스 (규모 언급 시), P6 유형별 질문 |
| 영역 4 (실패 시나리오) | P7 제약사항 (환경 제약이 실패 시나리오와 겹칠 수 있음) |

#### "충분" 판단 시 출력 형식

```
기획 인터뷰에서 [영역명] 관련 내용이 이미 나왔습니다:
- [기획 답변 요약]

이 내용으로 충분한가요? 추가할 부분이 있으면 말씀해주세요.
없으면 다음 영역으로 넘어갑니다.
```

### 동작

1. `references/design-questions.md` 참조
2. 4개 영역을 순서대로 진행 — 각 영역 시작 전 중복 제거 규칙 적용
3. 각 영역 답변 완료 후 다음 영역으로 이동

### 영역별 진행

**영역 1 — 파이프라인 흐름**

```
### 🔀 파이프라인 흐름

데이터/작업이 처음부터 끝까지 어떻게 흘러가는지 파악하겠습니다.

**1.** 전체 데이터/작업 흐름을 단계별로 설명해주실 수 있나요? (입력 → 처리 → 출력)
**2.** 각 단계에서 사용할 도구/서비스가 정해진 게 있나요?

모르시면 "추천해줘"라고 하시면 프로젝트에 맞는 흐름을 제안드립니다.
```

**영역 2 — 데이터 구조**

```
### 🗄️ 데이터 구조

저장해야 할 데이터의 구조를 파악하겠습니다.

**1.** 저장해야 할 주요 데이터 종류는 무엇인가요? (원본, 가공, 결과 등)
**2.** 각 데이터에 어떤 필드(컬럼)가 필요할 것 같나요?

모르시면 "추천해줘"라고 하시면 테이블과 컬럼을 제안드립니다. (각 컬럼의 필요 이유도 함께 설명합니다)
```

**영역 3 — 볼륨 및 갱신 주기**

```
### 📊 데이터 볼륨 & 갱신 주기

규모와 속도를 파악하겠습니다.

**1.** 데이터가 얼마나 빠르게 쌓이나요? (초당/분당/일당 건수)
**2.** 데이터 보관 기간은 어느 정도인가요?

대략적인 감만 있어도 충분합니다.
```

**영역 4 — 실패 시나리오**

```
### ⚠️ 실패 시나리오

이 시스템이 실패할 수 있는 상황을 미리 인식해두겠습니다.

**1.** 이 시스템이 실패할 수 있는 상황은 어떤 게 있을까요?

모르시면 "추천해줘"라고 하시면 유형에 맞는 대표 시나리오 3~5개를 제안드립니다.
(상세 대응은 개발 단계에서 작성합니다. 여기선 인식만 하면 됩니다.)
```

### 각 영역 완료 후

답변을 간단히 정리하고 다음 영역으로 넘어감:

```
✓ 파이프라인 흐름 정리 완료. 다음은 데이터 구조입니다.
```

## STEP 4 — 인터뷰 완료, 킥오프 문서 업데이트 및 안내

모든 인터뷰 완료 후 (또는 설계 인터뷰 패스 후):

### 킥오프 문서 업데이트

킥오프 문서(`{프로젝트명}_kickoff.md`)를 Read한 후 Edit으로 아래 내용을 갱신한다:

1. **YAML frontmatter** `last_skill`을 `kickoff-interview`로 변경

2. **Section 1 기본 정보** 테이블에 기획 인터뷰 답변 채움:
   - 목적 → P2 답변
   - 주요 사용자 (내부) → P1 답변
   - 주요 사용자 (외부 표기) → P1-b 답변 (포트폴리오 목적이고 P1-b를 수집한 경우)
   - 데이터 소스 → P3 답변
   - 결과 형태 → P4 답변

3. **Section 1 핵심 기능** 플레이스홀더를 P5 답변 기반 기능 테이블로 교체:
   ```markdown
   | # | 기능 | MVP 포함 |
   |---|---|---|
   | 1 | [P5 답변에서 나열] | ✅ / ⬜ 선택 |
   ```
   - **행 분리 규칙**: 하나의 기능에 필수/선택 구성 요소가 섞여 있으면 행을 분리한다 (`profile-template.md` 참조). 외부 서비스 의존성 표의 필수/선택 분류와 대응되어야 한다.

4. **Section 1 기술 스택** 플레이스홀더를 기술 테이블로 교체
   - 테이블 작성 후 "미정" 항목이 남아있는지 확인한다
   - "미정" 레이어가 있으면 사용자에게 확인:
     ```
     기술 스택에 아직 미정인 항목이 있습니다:
     - 프론트엔드: 미정
     - 백엔드: 미정
     - DB: 미정

     직접 지정하시겠습니까, 아니면 "추천해줘"라고 하시면 프로젝트에 맞는 기술을 제안드립니다.
     ```
   - "추천해줘" 시: 프로젝트 유형, 제약사항, 기존 기술 스택과의 호환성을 기반으로 추천 + 선택 이유 제시
   - 사용자가 "나중에 결정" 시: "미정"으로 유지 (허용)

5. **Section 1 제약사항** 플레이스홀더를 P7 답변 기반 제약사항 테이블로 교체

6. 설계 인터뷰 진행한 경우:
   - **Section 2 시스템 아키텍처** 플레이스홀더를 D1 답변 기반 내용으로 교체
   - **Section 3 데이터 구조** 플레이스홀더를 D2-D3 답변 기반 내용으로 교체
   - **Section 4 실패 시나리오** 플레이스홀더를 D4 답변 기반 내용으로 교체

7. 설계 인터뷰 패스한 경우:
   - Section 2, 3, 4를 통째로 제거 (헤더 포함)

### 안내 출력

설계 인터뷰 진행한 경우:

```
## ✅ 인터뷰 완료

**기획 인터뷰**: 완료
**설계 인터뷰**: 완료 (4개 영역 수집)

킥오프 문서가 업데이트되었습니다.
다음 단계로 `/kickoff-suggest`를 실행하면 AI가 추가 아이디어를 제안합니다.
```

설계 인터뷰 패스한 경우:

```
## ✅ 인터뷰 완료

**기획 인터뷰**: 완료
**설계 인터뷰**: 패스 (나중에 추가 가능)

킥오프 문서가 업데이트되었습니다.
다음 단계로 `/kickoff-suggest`를 실행하면 AI가 추가 아이디어를 제안합니다.
```

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-start`를 먼저 실행하도록 안내 |
| 사용자가 답변을 거부 | "패스" 의사를 확인하고, 해당 항목은 AI 추천으로 대체 제안 |
| 답변이 질문과 무관 | 질문을 다시 한 번 짧게 재표현하여 재질문 |
| 사용자가 중간에 중단 요청 | 수집된 내용까지 정리 후 안내 (부분 완료 가능) |

## 주의사항

- 질문을 한꺼번에 너무 많이 던지지 않음 (기획 최대 10개, 설계는 영역별 2~3개)
- "추천해줘" 시 반드시 근거와 함께 제시
- 설계 인터뷰에서 컬럼 추천 시 "왜 필요한지" 이유 필수
- 사용자 답변을 임의로 해석/확장하지 않음 — 모호하면 추가 질문
- 이 스킬은 킥오프 문서의 Section 1~4를 직접 채운다. 반드시 킥오프 문서를 Read 먼저 → 이후 Edit으로 플레이스홀더 교체
