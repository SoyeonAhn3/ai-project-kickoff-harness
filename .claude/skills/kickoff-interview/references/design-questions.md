# 설계 인터뷰 질문 가이드

## 진입 조건

- 기획 인터뷰(Step 3-1) 완료 후 진행
- 사용자가 "패스"하면 이 단계 전체를 건너뜀

## 4개 질문 영역

### 영역 1 — 파이프라인 흐름

데이터/작업이 소스에서 최종 사용자에게 도달하기까지의 단계별 흐름을 파악한다.

**기획 답변 중복 확인**: P3(데이터 소스), P4(결과 형태), P5(핵심 기능)에서 흐름이 이미 나왔는지 확인한다.
- 충분: 입력 → 처리 → 출력 흐름이 구체적으로 나왔으면 요약 확인 후 D1-3(외부 의존성)만 추가 질문
- 부분: 입출력은 나왔지만 중간 단계가 불분명하면 D1-1만 심화 질문
- 없음: 전체 질문

| # | 질문 | 수집 목적 |
|---|---|---|
| D1-1 | 전체 데이터/작업 흐름을 단계별로 설명해주실 수 있나요? (입력 → 처리 → 출력) | 아키텍처 개요 |
| D1-2 | 각 단계에서 사용할 도구/서비스가 정해진 게 있나요? | 기술 스택 확인 |
| D1-3 | 외부 서비스나 API에 의존하는 부분이 있나요? (인증, 결제, 알림, 클라우드 서비스 등) | 외부 의존성 파악 |

**"잘 모르겠어" 시 AI 추천 방식**:
- 프로젝트 유형 + 기획 답변을 기반으로 일반적인 파이프라인 흐름을 제안
- 외부 의존성은 프로젝트 목적에서 유추하여 "보통 이런 서비스를 연동합니다" 형태로 제안
- 예: "AI 이상탐지 → 일반적으로 수집(Kafka/파일) → 전처리(Python) → 모델추론 → 결과저장(DB) → 알림 구조입니다. 이 흐름이 맞나요?"

---

### 영역 2 — 레이어별 테이블/데이터 구조

각 레이어(Raw/Warehouse/Mart 등)에 어떤 테이블/데이터가 필요한지 파악한다.

**기획 답변 중복 확인**: P3(데이터 소스), P5(핵심 기능)에서 데이터 종류가 이미 나왔는지 확인한다.
- 충분: 데이터 소스와 종류가 구체적으로 나왔으면 요약 확인 후 D2-2(필드/컬럼)만 추가 질문
- 부분: 데이터 소스는 나왔지만 저장 구조는 불분명하면 D2-1만 심화 질문
- 없음: 전체 질문

| # | 질문 | 수집 목적 |
|---|---|---|
| D2-1 | 저장해야 할 주요 데이터 종류는 무엇인가요? (원본, 가공, 결과 등) | 데이터 레이어 설계 |
| D2-2 | 각 데이터에 어떤 필드(컬럼)가 필요할 것 같나요? | 스키마 초안 |

**"잘 모르겠어" 시 AI 추천 방식**:
- 프로젝트 목적에 맞는 테이블/컬럼을 추천
- 각 컬럼마다 "왜 필요한지" 한 줄 이유 제공
- 예:

```
[생산 이상탐지] 추천 테이블 구조:

1. raw_sensor_data (원본)
   - timestamp: 센서 측정 시각 → 시계열 분석의 기준
   - machine_id: 어떤 설비인지 구분 → 설비별 패턴 비교
   - value: 측정값 → 이상탐지 대상
   - sensor_type: 센서 종류 → 다중 센서 대응

2. anomaly_results (결과)
   - detected_at: 이상 감지 시각 → 알림 타이밍
   - machine_id: 대상 설비 → 조치 대상 특정
   - anomaly_score: 이상 점수 → 심각도 판단
   - root_cause: AI 분석 원인 → 조치 가이드

수정하거나 추가할 부분이 있나요?
```

### 영역 3 — 데이터 볼륨 및 갱신 주기

**기획 답변 중복 확인**: P3(데이터 소스)에서 규모나 주기가 이미 언급되었는지 확인한다.
- 충분: 규모와 주기가 구체적 수치로 나왔으면 요약 확인 후 건너뜀
- 부분: "많다/적다" 수준만 나왔으면 구체적 수치만 추가 질문
- 없음: 전체 질문

| # | 질문 | 수집 목적 |
|---|---|---|
| D3-1 | 데이터가 얼마나 빠르게 쌓이나요? (초당/분당/일당 건수) | 스케일 설계 |
| D3-2 | 데이터 보관 기간은 어느 정도인가요? | 스토리지 계획 |

**"잘 모르겠어" 시 AI 추천 방식**:
- 유사 프로젝트 사례 기반으로 일반적 규모 제시
- "정확하지 않아도 됩니다. 대략적인 크기만 알면 아키텍처를 잡을 수 있어요."

---

### 영역 4 — 실패 시나리오 / 엣지케이스

**기획 답변 중복 확인**: P7(제약사항)에서 실패 관련 내용이 이미 나왔는지 확인한다.
- 충분: 제약사항에서 실패 상황까지 구체적으로 나왔으면 요약 확인 후 건너뜀 (드문 경우)
- 부분: 환경 제약은 나왔지만 시스템 실패 시나리오는 없으면 D4-1 질문
- 없음: 전체 질문 (대부분 이 경우)

| # | 질문 | 수집 목적 |
|---|---|---|
| D4-1 | 이 시스템이 실패할 수 있는 상황은 어떤 게 있을까요? | 리스크 인식 |

**"잘 모르겠어" 시 AI 추천 방식**:
- 프로젝트 유형 기반으로 3~5개 대표 실패 시나리오 제시
- 예 (AI 프로젝트):
  1. 학습 데이터 품질 문제 (결측, 이상값)
  2. 모델 추론 지연 (실시간 요구사항 미충족)
  3. AI 판단 오류 (False Positive/Negative)
  4. 데이터 소스 연결 끊김
  5. 모델 드리프트 (시간 경과 시 성능 저하)

**주의**: 킥오프에서는 "인식" 수준만. 상세 시나리오는 개발 단계(test-scenario 스킬)에서 작성.

## 설계 인터뷰 운용 규칙

- **질문 순서**: 영역 1 → 2 → 3 → 4 순서대로 진행
- **묶음 방식**: 각 영역의 질문을 한 묶음으로 제시 (영역별 2개씩)
- **영역 간 이동**: 한 영역 답변 완료 후 다음 영역으로
- **추가 질문**: 답변이 모호하면 해당 영역에서 1개 추가 질문 가능
- **AI 추천 시**: "목적에 맞는 것"을 먼저 추천 (쉬운 것 X)
- **컬럼 추천 시**: 반드시 "왜 이 컬럼이 필요한지" 이유 포함
- **패스 허용**: 어떤 영역이든 "잘 모르겠어"로 AI 추천 채택 가능
