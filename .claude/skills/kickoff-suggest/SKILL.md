# kickoff-suggest Skill

인터뷰 결과를 분석하여 사용자가 생각하지 못한 유용한 기능/구조를 최대 5개 제안하는 스킬. "정리만 하는 도구"에서 "같이 기획하는 도구"로의 핵심 차별점.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/*_kickoff.md`를 검색
  - 0개 발견: `/kickoff-start`를 먼저 실행하도록 안내
  - 1개 발견: 해당 파일 Read → YAML frontmatter 확인
  - 2개 이상 발견: frontmatter에 `completed_skills` 필드가 있는 파일만 대상, 여러 개면 사용자에게 선택 요청
- YAML frontmatter `completed_skills`에 `kickoff-interview`가 포함되어야 함
- 킥오프 문서 Section 1에서 인터뷰 답변(핵심 기능, 제약사항 등) 확인, Sections 2-4에서 설계 데이터 확인 (있는 경우)

---

## STEP 1 — 컨텍스트 분석

대화 컨텍스트에서 아래 정보를 분석한다:

- 프로젝트 유형 (kickoff-start에서 확정)
- 기획 인터뷰 답변 (목적, 사용자, 핵심 기능, MVP 범위, 제약사항)
- 설계 인터뷰 답변 (진행한 경우: 아키텍처, 데이터 구조, 외부 의존성)

### 제안 생성 규칙

1. **목적에 맞는 것** 우선 (쉬운 것, 유행하는 것 X)
2. **사용자가 이미 언급한 기능은 제안하지 않음** (중복 방지)
3. **제약사항에 위배되는 제안 금지** (예: Docker 불가인데 컨테이너 기반 제안)
4. **최대 5개** (scope creep 방지)
5. **각 제안에 반드시 포함**: 왜 유용한지 + 난이도 + 예상 시간

### 컨텍스트 기반 필터링 규칙

킥오프 문서 `### 프로젝트 배경`(사용자 컨텍스트)이 존재하면 아래 필터링을 적용한다:

| 컨텍스트 항목 | 필터링 |
|---|---|
| 기술 수준: 초급 | 난이도 "높" 제안에 `⚠️ 현재 수준 대비 도전적` 경고 표시 |
| 시간 예산: 1~2주 | 예상 시간 3일 이상 제안에 `⚠️ 시간 예산 초과 가능` 경고 표시 |
| 목적: 포트폴리오 | "보여줄 수 있는" 기능 우선 (시각적 결과물, 데모 가능) |
| 목적: 학습 | "배움이 있는" 기능 우선 (새로운 기술/패턴 경험) |

- 경고가 있어도 채택 가능 — 사용자 판단 존중
- 컨텍스트 없으면 필터링 없이 기존 동작 유지

## STEP 2 — 아이디어 제안

### 출력 형식

```
## 💡 AI 추가 아이디어 제안

인터뷰 내용을 바탕으로, 프로젝트에 도움이 될 수 있는 아이디어를 제안합니다.
이 제안은 참고용이며, 채택하지 않아도 됩니다.

| # | 아이디어 | 왜 유용한가 | 난이도 | 예상 시간 |
|---|---|---|---|---|
| 1 | [제안 내용] | [1줄 근거] | 낮/중/높 | [시간] |
| 2 | [제안 내용] | [1줄 근거] | 낮/중/높 | [시간] |
| 3 | [제안 내용] | [1줄 근거] | 낮/중/높 | [시간] |
| 4 | [제안 내용] | [1줄 근거] | 낮/중/높 | [시간] |
| 5 | [제안 내용] | [1줄 근거] | 낮/중/높 | [시간] |

---

**어떻게 하시겠습니까?**
- 채택할 번호를 말씀해주세요 (예: "1, 3 채택")
- 수정해서 채택도 가능합니다 (예: "2번을 이렇게 바꿔서 채택")
- "전부 패스" — 제안 없이 다음 단계로 진행
```

### 난이도 기준

| 난이도 | 의미 | 예상 시간 범위 |
|---|---|---|
| 낮 | 기존 구조에 자연스럽게 추가 가능 | 수 시간 ~ 1일 |
| 중 | 별도 모듈/연동 필요 | 1~3일 |
| 높 | 아키텍처 변경 또는 복잡한 외부 연동 필요 | 3일 이상 |

## STEP 3 — 사용자 선택 처리

### 채택한 경우

```
## ✅ 채택된 아이디어

| # | 아이디어 | 분류 |
|---|---|---|
| 1 | [채택된 내용] | MVP / MVP 이후 |
| 3 | [채택된 내용] | MVP / MVP 이후 |

분류가 맞나요? 수정할 부분이 있으면 말씀해주세요.
```

### 분류 규칙

- 난이도 "낮" + 핵심 목적에 직결 → **MVP**
- 난이도 "중/높" 또는 부가적 기능 → **MVP 이후 확장**
- 사용자가 직접 분류를 지정하면 그것을 우선

### 수정 채택한 경우

사용자의 수정 내용을 반영하여 최종 아이디어를 정리하고, 동일하게 MVP/이후 분류.

### 전부 패스한 경우

```
## ✅ 제안 단계 완료

채택된 아이디어 없이 다음 단계로 진행합니다.
```

## STEP 4 — 완료, 킥오프 문서 업데이트 및 안내

### 킥오프 문서 업데이트

킥오프 문서(`{프로젝트명}_kickoff.md`)를 Read한 후 Edit으로 아래 내용을 갱신한다:

1. **YAML frontmatter** `last_skill`을 `kickoff-suggest`로 변경, `completed_skills`에 `kickoff-suggest` 추가

2. **Section 1 `### 채택된 AI 제안`** 플레이스홀더를 채택 테이블로 교체:

```markdown
### 채택된 AI 제안

| # | 아이디어 | 분류 |
|---|---|---|
| 1 | [채택된 내용] | MVP / 이후 확장 |
```

3. **Section 1 `### 핵심 기능`** 테이블에 MVP로 채택된 항목을 추가:
   - "MVP"로 분류된 채택 아이디어를 `### 핵심 기능` 테이블 하단에 행으로 추가 (✅ 표기)
   - "이후 확장"으로 분류된 항목은 추가하지 않음 (채택된 AI 제안 테이블에만 기록)
   - 번호는 기존 핵심 기능 마지막 번호 이후로 이어서 부여

4. 전부 패스 시: 플레이스홀더를 아래로 교체:

```markdown
### 채택된 AI 제안

채택된 아이디어 없음 (전부 패스)
```

### 안내 출력

```
## ✅ 아이디어 제안 완료

**채택**: [N]개 (MVP: [n]개, 이후 확장: [m]개)

킥오프 문서가 업데이트되었습니다.
다음 단계로 `/kickoff-profile`을 실행하면 킥오프 문서를 최종 검토합니다.
```

## 출력 형식

이 스킬은 **킥오프 문서를 직접 업데이트**한다:
- 대상: `{프로젝트명}_kickoff.md`
- 방식: Section 1 `### 채택된 AI 제안` 플레이스홀더 교체 + MVP 채택 항목을 `### 핵심 기능` 테이블에 추가 (Edit 사용)
- YAML frontmatter `last_skill` 및 `completed_skills` 갱신

---

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-start`를 먼저 실행하도록 안내 |
| 킥오프 문서에 인터뷰 데이터 없음 | `/kickoff-interview`를 먼저 실행하도록 안내 |
| 제안할 아이디어가 없음 (사용자가 이미 충분히 포괄적) | "현재 기획이 충분히 포괄적입니다"라고 안내 후 패스 처리 |
| 사용자가 5개 이상 추가 제안 요청 | "킥오프 단계에서는 5개로 제한합니다" 안내 |
| 채택 후 분류에 동의하지 않음 | 사용자가 지정한 분류로 재설정 |

## 주의사항

- 제안은 **강제가 아닌 참고용**임을 항상 명시
- "전부 패스"를 명시적 선택지로 항상 제공
- 이미 사용자가 기획한 기능과 중복되는 제안 금지
- 제약사항(P7 답변)에 위배되는 제안 금지
- 난이도와 시간은 정확하지 않아도 됨 — 대략적 감만 제공
- 이 스킬은 킥오프 문서의 Section 1 `### 채택된 AI 제안`을 직접 채운다. 반드시 킥오프 문서를 Read 먼저 → 이후 Edit으로 플레이스홀더 교체
