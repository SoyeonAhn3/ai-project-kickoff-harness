# 컨텍스트 전파 가이드

kickoff-context에서 수집한 사용자 메타 정보가 하위 스킬에 어떻게 영향을 주는지 정의한다.
모든 하위 스킬은 이 가이드를 참조하여 컨텍스트 데이터를 활용한다.

---

## 전파 원칙

1. **가이드이지 강제가 아님** — 컨텍스트는 스킬 동작의 힌트로 사용, 사용자 답변이 항상 우선
2. **없으면 기본 동작** — 컨텍스트 없이 실행해도 모든 스킬이 정상 동작해야 함
3. **읽기 위치** — 상태 파일 섹션 0-1에서 읽음 (kickoff-start가 생성)

---

## 스킬별 활용 규칙

### kickoff-start

- 상태 파일 생성 시 컨텍스트 데이터를 섹션 0-1로 포함
- 컨텍스트 없으면 섹션 0-1 생략

### kickoff-interview

| 목적 | 추가 질문 |
|---|---|
| 포트폴리오 | "이 프로젝트에서 가장 보여주고 싶은 기술적 포인트는?" |
| 업무 자동화 | "현재 수동 프로세스의 가장 큰 병목은?" |
| 학습 | "이전에 시도해봤지만 실패한 경험이 있나요?" |
| 실무 투입 | "SLA나 가용성 요구사항이 있나요?" |

- 기술 수준이 "초급"이면 설계 인터뷰에서 "추천해줘" 유도를 더 적극적으로
- 컨텍스트 없으면 추가 질문 없이 기존 동작 유지

### kickoff-suggest

| 컨텍스트 항목 | 필터링 규칙 |
|---|---|
| 기술 수준: 초급 | 난이도 "높" 제안에 ⚠️ 경고 표시 |
| 시간 예산: 1~2주 | 예상 시간 3일 이상 제안에 ⚠️ 경고 표시 |
| 목적: 포트폴리오 | "보여줄 수 있는" 기능 우선 제안 |
| 목적: 학습 | "배움이 있는" 기능 우선 제안 |

- 경고 표시 형식: `⚠️ 현재 수준/시간 대비 도전적일 수 있음`
- 경고가 있어도 채택은 가능 (사용자 판단 존중)
- 컨텍스트 없으면 필터링 없이 기존 동작 유지

### kickoff-evaluate (Phase 9)

- 컨텍스트가 평가의 핵심 입력 — 목적/수준/시간 대비 프로젝트 적절성 판단
- 목적별 "차별화"의 기준이 달라짐 (포폴: 채용 담당자 시선, 실무: 비즈니스 가치)
- 컨텍스트 없으면 일반적 평가만 수행 (목적 특화 불가)

### kickoff-gap

| 목적 | 추가 점검 |
|---|---|
| 포트폴리오 | README/데모 관련 내용이 있는지 확인 |
| 실무 투입 | 배포/모니터링 계획이 있는지 확인 |
| 웹/앱 + 외부 사용자 | 보안 누락 점검 (HTTPS, 인증, 입력 검증) |

- 컨텍스트 없으면 기본 5개 카테고리만 점검

### kickoff-checklist

| 목적 | 추가 체크 항목 |
|---|---|
| 포트폴리오 | README 초안 작성, 데모 시나리오 구상, 스크린샷/GIF 촬영 계획 |
| 업무 자동화 | 기존 프로세스 문서화, AS-IS/TO-BE 정리 |
| 실무 투입 | 배포 환경 확인, 모니터링 도구 선정, 롤백 절차 수립 |
| 웹/앱 + 외부 사용자 | 인증 방식 확정, SSL 인증서, 보안 점검 항목 확인 |

- 컨텍스트 없으면 공통 체크 항목만 생성

### kickoff-done (Phase 10)

| 목적 | 추가 완료 조건 카테고리 |
|---|---|
| 포트폴리오 | 데모 기준 (Demo-ready) |
| 학습 | 학습 증거 (Learning) |
| 실무 투입 | 운영 기준 (Operational) |
| 웹/앱 + 외부 사용자 | 보안 기준 (Security) |

- 컨텍스트 없으면 기본 3개(기능/품질/문서화)만 생성

---

## 보안 관련 활성화 조건

아래 두 조건을 **동시에** 만족할 때만 보안 관련 항목을 활성화한다:

1. 프로젝트 유형이 **웹/앱** (kickoff-start에서 판별)
2. 사용자가 **외부 사용자**를 대상으로 함 (인터뷰 또는 컨텍스트에서 확인)

**비활성 조건** (하나라도 해당하면 보안 비활성):
- 포트폴리오 + 더미 데이터만 사용
- 개인 학습용
- CLI 도구 (로컬 전용)
