# 완료 조건 템플릿

## 작성 원칙

1. **검증 가능** — "~한다", "~있다" (예/아니오로 판정 가능)
2. **구체적** — 수치, 상태, 파일명 등 명시 ("적절한" X → "3초 이내")
3. **MVP 한정** — "이후 확장" 기능은 포함하지 않음
4. **수준 맞춤** — 초급자에게 과도한 조건 지양

---

## 항상 적용 카테고리 (3개)

### 기능 동작 (Functional)

프로젝트의 핵심 기능이 정상 동작하는지 검증하는 조건.

**유형별 예시**:

| 프로젝트 유형 | 조건 예시 |
|---|---|
| 데이터 파이프라인 | "소스→싱크 데이터 흐름이 에러 없이 완료된다" |
| AI 프로젝트 | "모델 추론이 기대 형식의 결과를 반환한다" |
| 웹앱 | "핵심 유저 플로우(가입→로그인→주요 기능)가 정상 동작한다" |
| 자동화 | "스케줄 실행이 지정 시간에 정상 동작한다" |
| BI 대시보드 | "모든 차트가 데이터를 정상 표시한다" |
| CLI 도구 | "주요 명령어가 기대 결과를 출력한다" |
| 라이브러리 | "공개 API가 문서대로 동작한다" |

**생성 규칙**:
- MVP에 포함된 핵심 기능 각각에 대해 1개씩
- "에러 없이", "정상", "기대 결과" 등의 표현은 구체화 (어떤 에러? 어떤 결과?)

---

### 품질 기준 (Quality)

성능, 데이터 정합성, 에러 처리 등 비기능 요구사항.

**유형별 예시**:

| 프로젝트 유형 | 조건 예시 |
|---|---|
| 데이터 파이프라인 | "적재된 데이터의 행 수가 소스와 일치한다" |
| AI 프로젝트 | "정확도/재현율이 기준치 이상이다" |
| 웹앱 | "주요 페이지 로드가 3초 이내다" |
| 자동화 | "실패 시 알림이 발송된다" |
| BI 대시보드 | "데이터 갱신 주기에 맞게 업데이트된다" |

**생성 규칙**:
- 킥오프 문서의 제약사항/실패 시나리오에서 도출
- 수치가 없으면 합리적 기본값 제시 후 사용자 확인

---

### 문서화 (Documentation)

프로젝트를 다른 사람(또는 미래의 자신)이 이해할 수 있는 최소 문서.

**공통 조건**:
- "README에 프로젝트 설명과 실행 방법이 포함되어 있다"
- "환경 설정 방법(.env 등)이 문서화되어 있다"

**유형별 추가**:

| 프로젝트 유형 | 조건 예시 |
|---|---|
| 데이터 파이프라인 | "데이터 흐름도(텍스트 또는 다이어그램)가 존재한다" |
| AI 프로젝트 | "모델 선택 이유와 성능 지표가 기록되어 있다" |
| 라이브러리 | "공개 API 문서가 존재한다" |

---

## 조건부 카테고리 (4개)

### 데모 기준 (Demo-ready)

**활성화 조건**: 프로젝트 목적이 **포트폴리오**

**조건 예시**:
- "3분 이내 데모 시나리오가 존재한다"
- "스크린샷 또는 GIF가 README에 포함되어 있다"
- "실행 방법이 3단계 이내로 설명되어 있다"
- "데모용 샘플 데이터가 포함되어 있다"

**생성 규칙**:
- 채용 담당자가 README만 보고 프로젝트를 이해할 수 있는 수준
- "보여주기"에 초점 — 내부 코드 품질보다 외부 인상

---

### 운영 기준 (Operational)

**활성화 조건**: 프로젝트 목적이 **실무 투입**

**조건 예시**:
- "프로덕션 환경에 배포가 완료되어 있다"
- "health check 엔드포인트가 200을 반환한다"
- "모니터링 대시보드가 설정되어 있다"
- "롤백 절차가 문서화되어 있다"
- "로그 수집이 정상 동작한다"

**생성 규칙**:
- 배포 후 운영 가능한 상태인지 검증
- 장애 대응 가능 여부 포함

---

### 보안 기준 (Security)

**활성화 조건**: **웹/앱** 프로젝트 + **외부 사용자** 존재

**조건 예시**:
- "인증 없이 보호된 리소스에 접근이 불가능하다"
- "사용자 입력에 대한 검증(XSS, SQL injection)이 적용되어 있다"
- "민감 데이터(비밀번호 등)가 평문으로 저장되지 않는다"
- "HTTPS가 적용되어 있다"

**생성 규칙**:
- OWASP Top 10 기본 항목 기반
- 킥오프 문서에서 언급된 보안 요소 반영
- evaluate에서 보안 🟡/🔴가 있었으면 해당 항목 필수 포함

---

### 학습 증거 (Learning)

**활성화 조건**: 프로젝트 목적이 **학습**

**조건 예시**:
- "핵심 개념 3개 이상에 대해 자신의 말로 설명할 수 있다"
- "학습 내용 TIL 또는 블로그 포스트가 1개 이상 존재한다"
- "프로젝트를 통해 배운 점과 어려웠던 점이 기록되어 있다"

**생성 규칙**:
- "만들었다"가 아니라 "배웠다"를 증명하는 조건
- 결과물보다 과정 기록에 초점

---

## evaluate 🟡/🔴 대응 규칙

평가에서 지적된 항목은 완료 조건에 대응 항목이 반드시 포함되어야 한다.

| 평가 차원 | 🟡 대응 조건 예시 | 🔴 대응 조건 예시 |
|---|---|---|
| 차별화 | "README에 유사 프로젝트와의 차별점이 1줄 이상 명시되어 있다" | "차별화 요소가 핵심 기능에 반영되어 있고 데모로 확인 가능하다" |
| AI 적절성 | "AI/ML 모델의 역할과 한계가 문서화되어 있다" | "AI 모델이 규칙 기반 대비 개선된 결과를 보여주는 비교 데이터가 존재한다" |
| 시장 유효성 | "타겟 사용자/시장에 대한 1줄 설명이 README에 있다" | "시장 조사 결과 또는 사용자 피드백이 문서화되어 있다" |
| 완성도 기대치 | "MVP 전체 기능의 동작 데모가 존재한다" | "모든 MVP 기능이 엣지케이스 포함 테스트를 통과한다" |
| 학습 비용 | "핵심 기술에 대한 학습 기록이 존재한다" | "학습 로드맵 대비 실제 학습 진행률이 기록되어 있다" |
| 보안 적절성 | "기본 보안 체크리스트가 통과되었다" | "보안 테스트(자동 또는 수동) 결과가 문서화되어 있다" |

- evaluate 미실행 시 이 섹션은 건너뜀
- 🟢 항목은 대응 조건 불필요
