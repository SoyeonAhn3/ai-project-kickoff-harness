# kickoff-done Skill

킥오프 문서와 평가 결과를 기반으로, "이 프로젝트가 끝났다"고 말할 수 있는 조건 5~10개를 생성하는 스킬. 체크리스트가 "시작의 조건"이라면, 이 스킬은 "끝의 정의"를 만든다.

---

## 사전 조건

- 킥오프 문서(`{프로젝트명}_kickoff.md`)가 존재해야 함
- 대화 컨텍스트에 경로가 없으면 Glob으로 `**/kickoff_state.md`를 검색하여 Read → 섹션 0에서 산출물 경로와 프로젝트명 확인
- 상태 파일에서 `kickoff-evaluate` 완료가 아니면 `/kickoff-evaluate`를 먼저 실행하도록 안내
- evaluate 없이 실행해도 동작 (기본 3개 카테고리만 생성, 목적별 카테고리 비활성)

---

## STEP 1 — 입력 수집

상태 파일과 킥오프 문서를 읽어 완료 조건 생성에 필요한 정보를 수집한다.

### 수집 대상

| 출처 | 수집 항목 | 활용 |
|---|---|---|
| 상태 파일 섹션 0-1 | 프로젝트 목적, 기술 수준 | 조건부 카테고리 활성화 판단 |
| 킥오프 문서 섹션 1 | 유형, 핵심 기능, MVP 범위, 기술 스택 | 기능 동작 조건 생성 |
| 킥오프 문서 섹션 2~4 | 아키텍처, 데이터 구조, 실패 시나리오 | 품질 기준 조건 생성 |
| 킥오프 문서 섹션 7 | 정직한 평가 결과 (🟡/🔴 항목) | 🟡/🔴 대응 조건 필수 포함 |

### 조건부 카테고리 활성화 판단

| 조건부 카테고리 | 활성화 조건 | 판단 기준 |
|---|---|---|
| 데모 기준 | 포트폴리오 목적 | 상태 파일 섹션 0-1 |
| 운영 기준 | 실무 투입 목적 | 상태 파일 섹션 0-1 |
| 보안 기준 | 웹/앱 + 외부 사용자 | 유형 + 사용자 정보 |
| 학습 증거 | 학습 목적 | 상태 파일 섹션 0-1 |

- context 없으면 조건부 카테고리 전체 비활성 (기본 3개만 생성)

---

## STEP 2 — 카테고리별 완료 조건 생성

`references/done-criteria-templates.md`를 참조하여 프로젝트에 맞는 완료 조건을 생성한다.

### 카테고리 구조 (3+4)

**항상 적용 (3개)**:

| 카테고리 | 설명 |
|---|---|
| 기능 동작 (Functional) | MVP에 포함된 핵심 기능이 정상 동작하는지 |
| 품질 기준 (Quality) | 성능, 데이터 정합성, 에러 처리 등 |
| 문서화 (Documentation) | README, 실행 가이드, 설정 방법 등 |

**조건부 적용 (4개)**:

| 카테고리 | 활성화 조건 |
|---|---|
| 데모 기준 (Demo-ready) | 포트폴리오 목적 |
| 운영 기준 (Operational) | 실무 투입 목적 |
| 보안 기준 (Security) | 웹/앱 + 외부 사용자 |
| 학습 증거 (Learning) | 학습 목적 |

### 작성 규칙

1. **검증 가능한 문장** — "~한다", "~있다" 형태
2. **주관적 표현 금지** — "좋은", "적절한" X → 구체적 수치/상태
3. **총 5~10개** — 너무 적으면 모호, 너무 많으면 부담
4. **MVP 범위만** — "이후 확장"으로 분류된 기능은 포함하지 않음
5. **evaluate 🟡/🔴 대응 필수** — 지적된 항목에 대응하는 조건이 반드시 포함
6. **사용자 수준 맞춤** — context에 기술 수준이 있으면 난이도 조정

### evaluate 연동 규칙

킥오프 문서 섹션 7(정직한 평가)에서 🟡/🔴로 판정된 차원이 있으면:

| 평가 차원 | 대응 완료 조건 예시 |
|---|---|
| 차별화 🟡 | "README에 기존 유사 프로젝트와의 차별점이 명시되어 있다" |
| AI 적절성 🟡 | "AI 모델의 입출력과 성능 지표가 문서화되어 있다" |
| 학습 비용 🟡 | "핵심 기술 3개에 대한 학습 기록(TIL)이 존재한다" |
| 완성도 기대치 🟡 | "MVP 전체 기능이 에러 없이 동작하는 데모가 존재한다" |

- evaluate 미실행 시 이 규칙은 건너뜀
- 사용자가 "유지"를 선택했더라도 🟡/🔴 대응 조건은 포함 (참고용으로라도)

## STEP 3 — 결과 제시 및 사용자 확인

### 출력 형식

```
## 🏁 완료 조건 생성

project_kickoff.md를 기반으로, 이 프로젝트가 "끝났다"고 말할 수 있는 조건을 생성합니다.

### 기능 동작 기준
- [ ] [조건 1]
- [ ] [조건 2]

### 품질 기준
- [ ] [조건 3]

### 문서화 기준
- [ ] [조건 4]
- [ ] [조건 5]

### [목적별 추가 카테고리] (활성화된 경우)
- [ ] [조건 6]
- [ ] [조건 7]

---

수정/추가/삭제할 항목이 있으면 말씀해주세요.
없으면 project_kickoff.md에 반영합니다.
```

### 사용자 응답 처리

| 응답 | 처리 |
|---|---|
| "없어" / "좋아" | 그대로 반영, STEP 4로 |
| 항목 수정 요청 | 수정 후 다시 제시 |
| 항목 추가 요청 | 추가 후 다시 제시 |
| 항목 삭제 요청 | 삭제 후 다시 제시 (최소 5개 유지 안내) |

## STEP 4 — 파일 업데이트 및 상태 저장

### 킥오프 문서 섹션 8 작성

킥오프 문서의 섹션 8 플레이스홀더를 실제 내용으로 교체한다.

```markdown
## 8. 완료 조건 (Definition of Done)

> 아래 조건을 모두 만족하면 이 프로젝트는 "완료"입니다.

### 기능 동작 기준
- [ ] [조건]

### 품질 기준
- [ ] [조건]

### 문서화 기준
- [ ] [조건]

### [목적별 카테고리] (활성화된 경우)
- [ ] [조건]

생성일: [날짜]
```

### 상태 파일 갱신 (BL-001 슬림 패턴 적용)

상태 파일(`kickoff_state.md`)을 Read한 후 Edit으로 아래 내용을 추가/갱신한다:

1. `마지막 완료`를 `kickoff-done`으로 변경
2. `마지막 갱신`을 현재 시각으로 변경
3. 섹션 9를 추가:

```markdown
## 9. 완료 조건 (kickoff-done)

상태: 완료
조건 수: [N]개
활성 카테고리: [카테고리 목록]
evaluate 🟡/🔴 대응: [N]개 반영
```

### 안내 출력

```
## ✅ 완료 조건 생성 완료

**조건 수**: [N]개
**활성 카테고리**: [목록]
**evaluate 대응**: 🟡/🔴 [N]개 항목에 대응 조건 포함

`{프로젝트명}_kickoff.md` 섹션 8이 업데이트되었습니다.
진행 상태가 `kickoff_state.md`에 저장되었습니다.

🎉 킥오프가 완료되었습니다!
이제 `/kickoff-skills`로 기본 개발 스킬을 설치하고 개발을 시작할 수 있습니다.
```

## 출력 형식

이 스킬은 **기존 파일을 수정하고 상태 파일을 갱신**한다:
- 수정: `{산출물 경로}/{프로젝트명}_kickoff.md` 섹션 8
- 갱신: `{산출물 경로}/kickoff_state.md` 섹션 9 추가

## 실패 처리

| 실패 유형 | 처리 방법 |
|---|---|
| 킥오프 문서 없음 | `/kickoff-profile`을 먼저 실행하도록 안내 |
| 섹션 8 플레이스홀더 없음 | 파일 하단에 섹션 8을 새로 추가 |
| evaluate 미실행 | 기본 3개 카테고리만 생성 + "evaluate 실행 시 더 정밀한 조건 생성 가능" 안내 |
| 조건이 5개 미만 | 프로젝트 내용이 부족한 경우 — 최소 3개는 생성 후 사용자에게 추가 요청 |
| 사용자가 전부 삭제 요청 | "최소 3개는 유지하는 것을 권장합니다" 안내 |

---

## 주의사항

- 체크리스트(착수 조건)와 혼동 금지 — done은 "끝"의 정의, checklist는 "시작"의 정의
- 완료 조건이 너무 높으면 좌절, 너무 낮으면 무의미 — 사용자 수준에 맞게 조정
- "이후 확장"으로 분류된 기능은 완료 조건에 넣지 않음 (MVP 범위만)
- 반드시 킥오프 문서를 **Read 먼저** → 이후 Edit으로 추가
- 이미 섹션 8에 내용이 있으면 (재실행 시) 기존 내용을 교체
