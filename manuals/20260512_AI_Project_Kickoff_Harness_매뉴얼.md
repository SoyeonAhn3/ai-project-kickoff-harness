## 1. 개요

AI Project Kickoff Harness는 프로젝트 아이디어를 **구조화된 AI 인터뷰**를 통해 개발 가능한 킥오프 문서로 변환하는 도구다. Claude Code의 Skill 시스템 위에서 동작하며, 14개 슬래시 커맨드를 순서대로 실행하면 요구사항 수집부터 설계 문서까지 하나의 Markdown 파일에 자동 생성된다.

| 항목 | 내용 |
|---|---|
| 동작 환경 | Claude Code CLI (터미널, 데스크톱 앱, VS Code 확장) |
| 산출물 | `{프로젝트명}_kickoff.md` (단일 기준 문서) |
| 지원 언어 | 한국어 / English (킥오프 시작 시 선택) |
| 지원 프로젝트 유형 | AI, 자동화, BI/데이터분석, 파이프라인, 웹앱, CLI, 라이브러리 (7종) |
| 스킬 수 | v1 킥오프 10개 + v2 설계 4개 = 총 14개 |

## 3. 시작하기

1. 터미널에서 하네스 폴더로 이동
   ```
   cd AI-Project-Kickoff-Harness
   ```
2. Claude Code 실행
   ```
   claude
   ```
3. 슬래시 커맨드로 킥오프 시작
   ```
   /kickoff-start
   ```
4. 언어를 선택하고 프로젝트 아이디어를 자유롭게 설명
5. 이후 스킬을 순서대로 실행 (아래 [4. 주요 기능](#4-주요-기능) 참조)

### 실행 순서 요약

```
[v1 킥오프 — 기획]
/kickoff-context     → (선택) 사용자 배경 수집
/kickoff-start       → 아이디어 입력 + 유형 판별
/kickoff-interview   → 기획 인터뷰 + 설계 인터뷰
/kickoff-suggest     → AI 아이디어 제안
/kickoff-profile     → 킥오프 문서 최종 검토
/kickoff-evaluate    → 정직한 평가
/kickoff-done        → 완료 조건 생성

[v2 설계 — 상세 설계 (선택)]
/design-requirements → 요구사항 정의
/design-architecture → 시스템 아키텍처 상세
/design-data-model   → 데이터 모델
/design-ai-workflow  → AI 워크플로우 (AI 프로젝트만)

[마무리]
/kickoff-gap         → 누락/모순 점검
/kickoff-checklist   → 개발 착수 체크리스트
/kickoff-skills      → 프로젝트 폴더에 기본 스킬 복사
```

## 4. 주요 기능

### 4.1 kickoff-context — 사용자 컨텍스트 수집

프로젝트를 시작하기 전에 "누가, 왜 만드는지"를 파악하는 선택적 스킬이다. 수집된 컨텍스트는 이후 모든 스킬에 전파되어 사용자 맥락에 맞는 추천과 평가를 가능하게 한다.

| 수집 항목 | 예시 |
|---|---|
| 프로젝트 목적 | 포트폴리오 / 업무 자동화 / 학습 / 실무 투입 |
| 대상 독자/평가자 | 채용 담당자 / 팀장 / 본인 |
| 현재 기술 수준 | 초급~고급 + 구체 역량 |
| 목표 직무 | 직무명 또는 "없음" |
| 시간 예산 | 예상 개발 기간 |
| 차별화 방향 | 키워드 |

> ⚠️ **주의:** 선택적 스킬이므로 패스해도 이후 진행에 문제없음. 단, context가 있으면 평가/제안 품질이 향상됨.

---

### 4.2 kickoff-start — 아이디어 입력 + 유형 판별

자유 텍스트로 프로젝트 아이디어를 입력하면 7개 프로젝트 유형 중 하나(또는 복합)로 자동 판별한다. 프로젝트 폴더와 킥오프 문서를 생성하는 진입점 스킬.

| 단계 | 동작 |
|---|---|
| 언어 선택 | 한국어 / English 중 선택 (이후 모든 산출물에 적용) |
| 아이디어 입력 | 자유 텍스트로 설명 (5단어 이상) |
| 유형 판별 | 키워드 + 맥락 분석으로 자동 판별, 사용자 확인 |
| 프로젝트 설정 | 프로젝트명, 저장 경로 지정 (미입력 시 자동 생성) |
| 문서 생성 | `{프로젝트명}_kickoff.md` 초기 문서 생성 |

**지원 프로젝트 유형:** AI, 자동화, BI/데이터분석, 파이프라인, 웹앱, CLI, 라이브러리

---

### 4.3 kickoff-interview — 기획 & 설계 인터뷰

프로젝트 유형에 맞는 구조화된 질문을 통해 요구사항을 수집한다. 기획 인터뷰(What)와 설계 인터뷰(How)의 2단계로 구성.

| 인터뷰 | 내용 | 패스 가능 |
|---|---|---|
| 기획 (What) | 목적, 사용자, 핵심 기능, 데이터 소스, 결과 형태, 기술 스택, 제약사항 | 아니오 |
| 설계 (How) | 파이프라인 흐름, 외부 의존성, 데이터 구조, 실패 시나리오 | 예 |

- 질문은 3~8개씩 묶어서 제시 (부담 최소화)
- "잘 모르겠어" / "추천해줘" → AI가 즉시 대안 제시
- 유형별로 다른 질문 세트 적용

---

### 4.4 kickoff-suggest — AI 아이디어 제안

인터뷰 결과를 분석하여 사용자가 생각하지 못한 기능/구조를 최대 5개 제안한다.

| 항목 | 설명 |
|---|---|
| 제안 수 | 최대 5개 |
| 포함 정보 | 아이디어, 난이도, 예상 소요 시간, MVP 적합 여부 |
| 사용자 선택 | 각 제안을 채택/보류/거절 가능 |
| 채택 시 | 킥오프 문서 "채택된 AI 제안" 섹션에 자동 추가 |

---

### 4.5 kickoff-profile — 킥오프 문서 최종 검토

이전 스킬들(start → interview → suggest)이 채운 킥오프 문서의 일관성과 품질을 검토하는 스킬. 이 시점부터 킥오프 문서가 **Single Source of Truth**가 된다.

| 검토 항목 | 내용 |
|---|---|
| 일관성 | 유형 ↔ 기능 ↔ 기술 스택 간 모순 확인 |
| 완성도 | "미정" 항목 확인, 누락 섹션 체크 |
| 품질 | 핵심 기능 MVP 분류, 기술 스택 선택 이유 구체화 |

---

### 4.6 kickoff-evaluate — 정직한 평가

4+2개 차원으로 프로젝트의 가치와 실현가능성을 점수화하여 객관적으로 평가한다.

| 차원 | 항목 | 상시/조건부 |
|---|---|---|
| 가치 | 기술 성장 기여도 | 상시 |
| 가치 | 실용/비즈니스 가치 | 상시 |
| 가치 | 차별화 | 상시 |
| 실현가능성 | 기간 내 완성 가능성 | 상시 |
| 조건부 | 학습 비용 대비 효율 | 포트폴리오/학습 |
| 조건부 | 보안 리스크 | 웹/앱 + 외부 사용자 |

- 각 차원별 1~5점 + 근거 제시
- 점수가 낮은 차원에 대해 개선 방안 제안
- MVP 범위 조정 시 문서 연쇄 갱신 체크리스트 제공

---

### 4.7 kickoff-done — 완료 조건 생성

"이 프로젝트가 끝났다"고 말할 수 있는 측정 가능한 조건 5~10개를 생성한다.

| 카테고리 | 예시 |
|---|---|
| 핵심 산출물 | "API 엔드포인트 N개 동작" |
| 품질 기준 | "테스트 커버리지 80% 이상" |
| 문서 | "README에 실행 방법 포함" |
| 배포 (조건부) | "Docker 이미지 빌드 성공" |

---

### 4.8 design-requirements — 요구사항 정의 (v2)

킥오프 문서(섹션 1~8)를 분석하여 기능 요구사항(FR), 비기능 요구사항(NFR), 사용자 흐름을 생성한다.

| 산출물 | 내용 |
|---|---|
| FR 테이블 | FR-001~ 형식, 핵심 기능에서 도출, 우선순위 포함 |
| NFR 테이블 | NFR-001~ 형식, 7개 카테고리 조건부 활성화 |
| 사용자 흐름 | 주요/대안/예외 흐름 |
| 갭 인터뷰 | 불명확한 요구사항 최대 5개 질문 |

**NFR 카테고리:** 성능, 보안(조건부), 확장성(조건부), 가용성(조건부), 유지보수성, 호환성(조건부), 데이터 품질(조건부)

---

### 4.9 design-architecture — 시스템 아키텍처 상세 (v2)

섹션 9(요구사항)를 기반으로 시스템 구조, 기술 선택 근거, 배포 구성, 아키텍처 결정 기록(ADR)을 생성한다.

| 산출물 | 내용 |
|---|---|
| 시스템 다이어그램 | 텍스트 기반 구조도 |
| 컴포넌트 테이블 | 역할, 기술, 입출력 |
| 기술 선택 근거 | 비교 대안 포함 |
| 배포 구성 | 조건부 (실무/외부 서비스만) |
| ADR | 트레이드오프가 있는 결정에만 작성 |

---

### 4.10 design-data-model — 데이터 모델 (v2)

엔티티 정의, 관계, 정합성 규칙, 데이터 생명주기, 마이그레이션 전략을 생성한다.

| 산출물 | 내용 |
|---|---|
| 엔티티 정의 | 필드, 타입, 제약 조건 |
| 관계 | 엔티티 간 1:N, N:M 등 |
| 정합성 규칙 | 비즈니스 규칙 + NFR 기반 |
| 마이그레이션 | 조건부 (실무 투입만) |

---

### 4.11 design-ai-workflow — AI 워크플로우 (v2)

AI 프로젝트에서만 활성화되며, AI 입출력 정의, 프롬프트 설계, 모델 선택, 폴백 전략을 생성한다. 비AI 프로젝트는 자동 스킵.

| 산출물 | 내용 |
|---|---|
| AI 입출력 정의 | 기능별 입력/출력 형식, 검증 규칙 |
| 프롬프트 설계 | 시스템 프롬프트, 템플릿, Temperature |
| 모델 선택 | 비용 추정 포함 |
| 폴백 전략 | 실패 시나리오별 대체 동작 |
| 평가/모니터링 | AI 품질 지표 + 목표치 |

---

### 4.12 kickoff-gap — 누락/모순 점검

킥오프 문서 전체를 스캔하여 누락된 정보, 섹션 간 모순, 실현 불가능한 조합을 찾아낸다.

| 점검 카테고리 | 내용 |
|---|---|
| 1. 필수 항목 누락 | "미정" 남은 항목, 빈 섹션 |
| 2. 교차 모순 | 기능 ↔ 제약 ↔ 기술 스택 간 충돌 |
| 3. 실현가능성 | 시간 예산 vs 기능 범위 |
| 4. 외부 의존성 | 필수 서비스 접근 가능 여부 |
| 5. 완료 조건 | 완료 조건 ↔ 핵심 기능 매핑 |
| 6. 설계 인터뷰 정합성 | 섹션 2~4 내부 일관성 |
| 7. v2 설계 교차 정합성 | NFR↔제약, 흐름↔아키텍처, 아키텍처↔데이터 모델, v1↔v2 |

---

### 4.13 kickoff-checklist — 개발 착수 체크리스트

킥오프 문서와 설계 문서를 기반으로 개발 시작 전 필요한 체크리스트를 생성한다. v2 설계 완료 시 프로젝트 구조 트리와 `.env.example`도 함께 생성.

| 산출물 | 조건 |
|---|---|
| 환경 설정 체크리스트 | 상시 |
| 도구/계정 체크리스트 | 상시 |
| 프로젝트 구조 트리 | v2 설계 완료 시 |
| `.env.example` | v2 설계 완료 시 |

### 4.14 kickoff-skills — 기본 스킬 복사

킥오프가 완료된 프로젝트 폴더에 GitHub Skill_package 레포에서 기본 개발 스킬 7개를 복사한다. `gh` CLI를 사용하여 전체 clone 없이 필요한 스킬만 선택 복사.

## 5. 데이터 저장 방식

- **단일 파일**: 모든 스킬의 산출물이 `{프로젝트명}_kickoff.md` 하나에 축적됨
- **YAML frontmatter**: 파일 상단에 프로젝트명, 생성일, 언어, 마지막 실행 스킬을 메타데이터로 저장
  ```yaml
  ---
  project: my_project
  created: 2026-05-12
  language: ko
  last_skill: kickoff-done
  ---
  ```
- **진행 추적**: `last_skill` 필드로 어디까지 진행했는지 자동 추적
- **Revision History**: v2 설계 스킬이 기존 섹션을 수정할 때 변경 이력을 문서 하단에 기록
- **산출물 경로**: `../{프로젝트명}/pre-requirement/{프로젝트명}_kickoff.md` (하네스 상위 디렉토리)

---

## 6. 주의사항 및 제한사항

> ⚠️ **실행 순서 필수**: 각 스킬은 선행 스킬 완료를 확인한다. 순서를 건너뛰면 선행 스킬을 먼저 실행하라는 안내가 나온다.

> ⚠️ **킥오프 문서 수동 편집 주의**: 섹션 헤더(`## N. 제목`) 형식을 변경하면 이후 스킬이 파싱에 실패할 수 있다. 내용은 자유롭게 수정 가능하나 헤더 형식은 유지할 것.

> ⚠️ **v2 설계 스킬은 선택**: v1 킥오프(start~done)만으로도 개발 착수 가능. v2 설계(requirements~ai-workflow)는 더 상세한 설계가 필요할 때만 실행.

> ⚠️ **gh CLI 인증 필요**: `/kickoff-skills` 실행 전에 `gh auth login`으로 GitHub 인증 완료 필요. 미인증 시 스킬 복사 실패.

> ⚠️ **대화 컨텍스트 유지**: 하나의 Claude Code 세션에서 여러 스킬을 연속 실행하는 것이 권장됨. 세션이 바뀌면 킥오프 문서를 다시 읽어야 하므로 `last_skill`로 진행 상태를 자동 복원.

## 7. 문제 해결 (FAQ)

**Q. 킥오프 문서를 찾을 수 없다는 오류가 나옵니다.**
A. 스킬은 `**/*_kickoff.md` 패턴으로 파일을 검색한다. `/kickoff-start`를 먼저 실행하여 문서를 생성하거나, 파일명이 `_kickoff.md`로 끝나는지 확인.

**Q. "선행 스킬을 먼저 실행하세요"라고 나옵니다.**
A. YAML frontmatter의 `last_skill` 필드로 진행 상태를 확인한다. 안내된 스킬을 먼저 실행하면 된다.

**Q. 이미 작성한 섹션을 다시 만들고 싶습니다.**
A. 해당 스킬을 다시 실행하면 "덮어쓸까요, 수정할까요?" 선택지가 나온다. 덮어쓰기를 선택하면 해당 섹션만 새로 생성.

**Q. 프로젝트 유형이 잘못 판별되었습니다.**
A. `/kickoff-start` STEP 3에서 "아니요"라고 답하면 원하는 유형을 직접 지정할 수 있다.

**Q. v2 설계 스킬 없이 바로 개발을 시작해도 되나요?**
A. 가능하다. `/kickoff-done` 이후 `/kickoff-gap` → `/kickoff-checklist` → `/kickoff-skills`를 실행하면 v2 설계 없이도 체크리스트와 스킬이 생성된다.

**Q. 영어로 진행하고 싶습니다.**
A. `/kickoff-start` 실행 시 언어 선택에서 "2. English"를 선택하면 이후 모든 산출물과 질문이 영어로 출력된다.
